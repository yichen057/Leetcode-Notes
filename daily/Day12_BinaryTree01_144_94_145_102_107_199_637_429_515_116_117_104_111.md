## 题144.  Definition for a binary tree node

本题是前序

```python
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left # 左指针, 默认为空
#         self.right = right # 右指针, 默认为空
# 递归法三步走: 1) 确定递归函数的参数和返回值; 本题无返回值, 结果放参数里了, 参数表示每一层递归需要处理的当前的节点
#             2) 确定终止条件: 如果 node 为 None，则停止。如果你传的是数值，数值无法判断是否为“空节点”。
#             3) 确定单层递归逻辑： 在每一层取当前节点的 val，然后把左右孩子（作为新的节点）交给下一层处理。
class Solution:
    def preorderTraversal(self, root: Optional[TreeNode]) -> List[int]:
        result = []

        # dfs函数传递的参数是节点而不是数值. 参数： 每一层递归需要处理的是“当前的节点”
        # 如果只传了 node.left.val（一个具体的数字），下一层递归就再也找不到它的左右子节点了，递归就此中断。
        def dfs(node): # node: 代表当前正在处理的整棵树的根节点
            if node is None:
                return
            
            # 前序的顺序: 中左右
            result.append(node.val) # node.value: 获取当前这棵树根节点的值
            dfs(node.left) # node.left: 代表当前节点的左子树
            dfs(node.right)
        # 调用函数
        dfs(root)
        return result
```

2. 为啥获取value不是.value而是.val呢

这主要不是由 Python 语言决定的，而是由**你正在使用的二叉树类（TreeNode）是如何定义的**决定的。

在 LeetCode、数据结构教材以及大多数算法面试题中，二叉树节点的定义通常如下：

Python

```
class TreeNode:
    def __init__(self, val=0, left=None, right=None):
        self.val = val    # 这里定义了属性名是 val
        self.left = left
        self.right = right
```

以下是为什么通常用 `.val` 而不是 `.value` 的几个原因：

### 1. 约定俗成（Naming Convention）

在算法界，为了书写简洁，开发者习惯使用缩写：

- **`val`** 代替 `value`（数值）
- **`curr`** 代替 `current`（当前指针）
- **`res`** 代替 `result`（结果）

这就像在数学里我们习惯用 $x$ 和 $y$ 而不是 `unknown_number_one` 一样，是一种**行业习惯**。

### 2. LeetCode 的标准

由于大部分人学习二叉树是通过 LeetCode（力扣）等平台，而这些平台的官方定义里统一使用了 `self.val`。为了让代码在平台上能跑通，大家都会遵循这个命名。如果你在代码里写 `.value`，解释器会报错：

> ```
> AttributeError: 'TreeNode' object has no attribute 'value'
> ```

### 3. 如何判断该写哪个？

其实，在实际工作中，写 `.val` 还是 `.value` 取决于**定义这个类的人**。

- 如果定义里写的是 `self.value = x`，你就得用 `.value`。
- 如果定义里写的是 `self.data = x`，你就得用 `.data`。

3. python语言里, 递归时dfs(node.left)为啥没有.val

在 Python 的递归实现中，`dfs(node.left)` 后面不需要加 `.val`，主要原因在于**函数参数的职责**和**递归的设计逻辑**：

### 1. 递归传递的是“节点”，而不是“数值”

在 `preorderTraversal` 逻辑中，`dfs` 函数定义的参数是 `node`。

- **`node` 的含义：** 它代表当前正在处理的整棵子树的根节点。
- **`node.left` 的含义：** 它代表当前节点的左子树（这也是一个 `TreeNode` 对象）。
- **为什么要传节点：** 递归需要不断深入到下一层，下一层依然需要知道这个节点的 `val`、`left` 和 `right`。如果你只传了 `node.left.val`（一个具体的数字），下一层递归就再也找不到它的左右子节点了，递归就此中断。

### 2. 数值的提取是在函数内部完成的

观察代码的执行流程：

1. 你调用 `dfs(node.left)`。
2. 程序跳转到 `dfs` 函数内部。
3. 此时，**当前的** `node` 已经变成了刚才传进来的左孩子。
4. 在这一层函数里，代码会通过 `res.append(node.val)` 提取出该节点的数值。

### 3. 符合递归的三要素

根据页面中提到的“递归三要素”：

- **参数：** 每一层递归需要处理的是“当前的节点”。
- **终止条件：** 如果 `node` 为 `None`，则停止。如果你传的是数值，数值无法判断是否为“空节点”。
- **单层逻辑：** 在每一层取当前节点的 `val`，然后把左右孩子（作为新的节点）交给下一层处理。

### 总结

- `dfs(node.left)`：意思是“去处理左边的那棵树”。
- `node.val`：意思是“获取当前这棵树根节点的值”。

如果写成 `dfs(node.left.val)`，就会把“一棵树”错当成“一个数字”传给函数，导致程序报错（因为数字没有 `.val` 属性，也无法进行后续的递归）。



4. void代表啥

在编程（特别是你正在看的 C++、Java 或 C# 代码）中，`void` 的核心含义是 **“空”** 或 **“无”**。

在二叉树遍历的递归函数中，你经常会看到 `void traversal(TreeNode* root)`，这具体代表以下几层意思：

### 1. 函数没有返回值

当一个函数的返回类型被标注为 `void` 时，意味着这个函数执行完操作后，**不会向调用它的地方返回任何数据**。

- **对比：** 如果是 `int sum()`，它必须最后 `return` 一个整数；如果是 `void`，它只需要完成它的任务（比如打印节点、把值存入数组）然后结束即可。

### 2. 在递归中的作用

在你查看的这段[二叉树递归遍历视频](https://www.bilibili.com/video/BV1Wh411S7xt/)中，`void` 非常常见。因为递归遍历通常有两种处理数据的方式：

- **直接打印/操作：** 函数只负责走遍每个节点，并在节点上做动作，不需要返回结果。
- **操作全局/引用变量：** 比如你在函数外定义了一个 `vector`（动态数组），递归函数通过引用的方式把节点值放进去。这时候函数本身不需要返回这个数组，所以用 `void`。

### 3. 语法细节

- **Return 语句：** 在 `void` 函数里，你可以写 `return;`（后面不跟任何值），这通常用于**提前结束函数**。

- **递归终止：** 在二叉树递归中，最常见的写法是：

  C++

  ```
  if (root == NULL) return; // 遇到空节点直接返回，不回传任何值
  ```

------

**简单来说：** `void` 就像是一个“只干活，不回话”的指令。

## 题 94: 二叉树的中序遍历

本题是中序

```python
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right
# 二叉树的中序遍历: 顺序是左中右
class Solution:
    def inorderTraversal(self, root: Optional[TreeNode]) -> List[int]:
        result = []

        def dfs(node):
            if node is None:
                return
            
            dfs(node.left)
            result.append(node.val)
            dfs(node.right)
        
        dfs(root)
        return result
```

# Definition for a binary tree node.

# 题102: 二叉树的层序遍历

```python
# @lc code=start
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right
# 上述为声明了一个类TreeNode, 并初始化做了构造函数, 定义了节点产生时的默认状态:  val=0 /left=None /right=None

# 如果使用的是 `collections.deque`，记住这四个最核心的操作就够了：
# - **入队（尾部）**: `queue.append(node)`
# - **出队（头部）**: `node = queue.popleft()`  （**这就是你问的那个 `cur`**）
# - **查看队首（不弹出）**: `first_node = queue[0]`
# - **查看队尾（不弹出）**: `last_node = queue[-1]`
# 层序遍历法: 是一层一层地“扫描”，先扫完第一层，再扫第二层
class Solution:
    def levelOrder(self, root: Optional[TreeNode]) -> List[List[int]]:
    # root 是 TreeNode 类的一个实例（Instance）, 虽然它叫 root（根），但它本质上和你后面遍历到的 cur、node 是同一种东西，只是因为它排在最前面，是我们抓取整棵树的“入口”，所以给了它一个特殊的名字叫 root
    # root有三个属性: 1) 数据 (val)：比如数字 1; 2)左指针 (left)：指向左下方的另一个盒子（对象）; 3)右指针 (right)：指向右下方的另一个盒子（对象）
    # root是变量名。 Optional[TreeNode]: 这是类型提示（Type Hint）。它的意思是：root 这个变量，要么是 TreeNode 类型的对象，要么是 None（空）。它绝对不是一个整数（int），也不是一个列表（list）
        if not root:# 判断root是否为空, 还可以写为: if root is None:明确判断是否为None
            return []
        queue = collections.deque([root]) # queue 队列：它是一个“缓冲区”。当你把第n层的节点一个个从队首取出时，你同时把它们的第 n+1 层孩子节点一个个塞进队尾
        # 虽然deque功能多（它是双端的），但因为它做“队首弹出”这个动作比list快太多了，所以 Python 程序员约定俗成: 只要是在 Python 里写算法题需要用到队列，统统使用collections.deque
        # Python 有一个库叫 queue，里面有个类叫 Queue, 不用它是因为: 它主要是给多线程编程用的。为了保证多线程安全，它加了很多“锁”的机制，导致它运行起来非常慢。在 LeetCode 这种单线程算法题里，用它是累赘
        # 你在做什么：你在实现一个队列（Queue）。你用的什么：你用的是双端队列（Deque）。为什么这么写：因为 Python 的 deque 是实现队列最快的方式，我们就只用它的 append（尾部进）和 popleft（头部出）这两个功能，假装它是一个普通的队列
        result = []
        while queue:
            level = [] # level 数组：它的生命周期非常短，只负责收集当前这一层（第n层）所有节点的值。一旦这一层循环结束，level 就会被装进结果大列表 res 中，然后被清空重置。
            for _ in range(len(queue)): # 在 Python 中，不能写 while(size--). Python 不支持 -- 或 ++ 运算符, 在 Python 中，必须明确写成 size -= 1。
# range 在进入循环前，对 len(queue) 做了一次“快照”（Snapshot), 先计算参数len(range), 然后生成迭代器, 然后循环的次数就在这一刻被锁定了
                # range() 是由 C 语言实现的优化迭代器，通常比手动维护一个 while 计数器稍微快一点. 用python写while的话: while size>0: ..... size -= 1
                cur = queue.popleft() # 取值+出队一行搞定; Python 的 deque 没有 front() 方法; TreeNode 对象没有 pop 方法
                level.append(cur.val) # level.append(...)存储的是数字
                if cur.left: # 如果左孩子不为空
                    queue.append(cur.left) # 当前节点cur和它的左右孩子都不在同一层, cur.left 和 cur.right 是第 N+1 层的节点。它们不能混进第 N 层的 level 结果里。它们必须去 queue（候车室） 排队，等待下一轮 for 循环（下一层）开始时，才能变成主角。
                    # 进 level：是因为它是现在的成果（数值）; 进 queue：是因为它是未来的任务（对象），需要排队等下一轮处理。
                    # queue.append(...): 必须存完整的节点对象（TreeNode）。因为只有存了对象，下一轮我们才能通过它访问 node.val、node.left 和 node.right
                if cur.right:
                    queue.append(cur.right)
            result.append(level)
        return result

# 递归法
#在递归过程中，虽然程序是“一路深钻”到底的（比如先处理完最左边的一条线），但由于我们随身携带了 level 这个深度信息，每到一个新节点，我们都能通过 levels[level] 准确地把它归类到它所属的那一层去。
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right
class Solution:
    def levelOrder(self, root: Optional[TreeNode]) -> List[List[int]]:
        if not root:
            return []
        
        levels = [] # levels 是一个二维列表。它的**索引（Index）**代表树的深度（层数）。它的元素是另一个列表，存放着对应层级的所有节点值。例如：levels[0] 存放第一层的值，levels[1] 存放第二层的值。
        
        # 定义在函数内部，不需要 self. 如果是写在 levelOrder 外面（作为类的方法） -> 必须加 self
        # 这里的 node 和 level 没写类型，完全没问题！# Python 解释器会自动处理，只要你传对了就行。def traverse(node, level):
        def traverse(node: Optional[TreeNode], level:int) -> None:
        # 参数 level 是一个整数, 用来访问列表levels的下标(索引),记录了当前递归到了哪一层。这是递归法能“找对位置”的关键; 
        # node 通常是 Tree Node 类的实例, 它是一个object对象, 负责提供数据和路径, 有以下属性：node.val（数据），node.left（左指针），node.right（右指针）,它不是数字索引。你不能用它来做加减法（比如你不能写 node + 1）
# Optional[TreeNode] 在 Python 的 typing 模块里的定义其实等同于： Union[TreeNode, None] （翻译：这个参数要么是 TreeNode，要么是 None）
# 如果当前的 node 是一个叶子节点（比如它下面没有左孩子了），那么 node.left 的值就是 None。 当你把 None 传给 traverse 函数时，参数 node 就接收到了 None。

# 如果你写 node: TreeNode，意思是：“我保证传进来的永远是一个真正的节点对象”。

# 如果你写 node: Optional[TreeNode]，意思是：“传进来的可能是个节点对象，但也可能是 None”。

# 显然，后者才符合事实
            if not node:
                return
            if len(levels) == level: # 当你第一次进入一个新的深度时，levels 的长度会等于当前的 level。此时，代码会往 levels 里添加一个空列表，为这一层“开辟空间”。
                levels.append([])

            levels[level].append(node.val) # 直接利用 level 作为索引，把当前节点的值放进对应层级的列表里。
            traverse(node.left, level+1) # 向左递归
            traverse(node.right, level +1) # 向右递归
        
        # 调用时也不需要 self
        traverse (root, 0)
        return levels

在 Python 中，`if not root:` **确实是用来判断 `root` 是否为空（或等价于“假”的值）**。

在二叉树这类算法题中，它最直接的含义就是：**如果当前节点（root）是 `None`，则执行下方的代码。**

------

### 1. 为什么可以这样写？

Python 的 `if` 语句会自动对变量进行“布尔测试”。以下值在 Python 中会被视为 **`False`**（假）：

- `None`（最常见的情况，表示空节点）
- `False`
- 数值 `0`
- 空字符串 `""`
- 空列表 `[]`、空字典 `{}` 等

在 [二叉树的层序遍历](https://programmercarl.com/0102.二叉树的层序遍历.html#_102-二叉树的层序遍历) 的代码中，`root` 通常是一个对象。如果它是 `None`，那么 `not root` 就会变成 `True`。

### 2. 常见的代码写法对比

在处理树节点时，以下几种写法在逻辑上通常是等价的，但 `if not root` 最为简洁：

| **写法**           | **风格**             | **语义**             |
| ------------------ | -------------------- | -------------------- |
| `if not root:`     | **推荐（Pythonic）** | 如果 root 为空       |
| `if root is None:` | **更严谨**           | 明确判断是否为 None  |
| `if root == None:` | 不推荐               | 性能略低于 `is None` |

### 3. 在算法中的实际应用

在你提供的 [Python 模板代码](https://programmercarl.com/0102.二叉树的层序遍历.html#python) 中：

Python

```

我们可以把这个逻辑总结为一句话：**`level` 是“当下”，而 `queue` 是“未来”。**

### 1. 为什么它们不在一层？

二叉树的层序遍历（BFS）核心逻辑就是：当你处理第 n  层的节点时，它的孩子节点必然属于第 n+1 层。

- **`level` 数组**：它的生命周期非常短，只负责收集**当前这一层**（第 n层）所有节点的值。一旦这一层循环结束，`level` 就会被装进结果大列表 `res` 中，然后被清空重置。
- **`queue` 队列**：它是一个“缓冲区”。当你把第 n层的节点一个个从队首取出时，你同时把它们的第 n+1 层孩子节点一个个塞进队尾。

### 2. 空间上的错位

想象一个具体的场景：

- 当你正在处理第一层（根节点 `root`）时：
  - 你的 `for` 循环只运行 1 次。
  - `level` 存入 `root.val`。
  - `queue` 存入 `root.left` 和 `root.right`（它们是第二层的）。
- 当这一层 `for` 循环结束，`level` 功成身退。
- 下一轮 `while` 循环开始时，`queue` 里的内容恰好就是**整整一层的第二层节点**。

### 3. 如果误用了 `level.append(cur.left)` 会发生什么？

如果你真的写了 `level.append(cur.left)`，会出现两个严重的错误：

1. **数据类型错误**：`level` 应该是存数字的（比如 `[1, 2, 3]`），结果你存进去了节点对象（`TreeNode`），输出结果就不对了。
2. **逻辑断层**：最关键的是，你没有把孩子放到 `queue` 里。这意味着当你处理完当前层后，`queue` 就空了，你的 `while queue` 循环会直接停止，**整棵树剩下的部分你就再也找不到了**。

### 总结

- **`level.append()`**：只放 **`cur.val`**（当前层的值）。
- **`queue.append()`**：只放 **`cur.left/right`**（下一层的节点，留着待会儿处理）。

### 1. `range()` 的“快照”机制

Python 的 `for i in range(N)` 语句在执行的那一瞬间，遵循以下逻辑：

1. **先计算参数**：解释器会**立即**执行 `len(queue)`，拿到一个具体的数字（比如 3）。
2. **生成迭代器**：然后生成一个固定长度的迭代器（比如 `0, 1, 2`）。
3. **锁定次数**：循环的次数就在这一刻被**锁定**了。

即便你在循环体内部疯狂地 `queue.append(...)`，导致 `len(queue)` 变成了 100，这个 `for` 循环也只会执行 **3次**，因为它只认准了**一开始**生成的那个范围。

for _ in range(size): # 2. 这里的 size 是个死数字，不会变
    cur = queue.popleft()
    # append 子节点... 
```

#### 写法 B：Python 简洁写法（当前代码）

Python

```
# len(queue) 在这行代码运行时被求值一次，变成了数字（比如3）
# range(3) 生成了，循环次数锁死为 3
for _ in range(len(queue)): 
    cur = queue.popleft()
    # append 子节点...
```

**结论：** 写法 B 就是写法 A 的“语法糖”，Python 帮你省去了定义 `size` 变量这一步，但在逻辑上它确实是**先取了 size**。

------

### 3. 什么时候会出问题？（危险写法）

你的担心在某些特定的写法下**确实会发生**。如果我们不用 `range`，而是每次循环都重新去查队列长度，那就完蛋了。

#### ❌ 错误写法（动态检查长度）

如果你用的是类似 C 语言风格的动态判断（虽然 Python 不能直接这样写 `for`，但假设用 `while` 模拟）：

Python

```
# 假设 queue 里原本有 1 个 root
# 危险！不要这样写层序遍历！

i = 0
while i < len(queue): # 每一次循环，都会重新计算 len(queue)！
    cur = queue.popleft() # 取出一个
    queue.append(cur.left) # 加入一个
    queue.append(cur.right) # 加入一个
    # 此时 len(queue) 变大了！循环永远停不下来！
    i += 1
```

在这个错误写法中，因为 `len(queue)` 每次循环都会**重新计算**，导致刚加进去的孩子节点，马上就在**这一轮**被当作当前层节点处理了，层级彻底乱套。

------

### 总结

你现在的代码：

Python

```
for _ in range(len(queue)):
```

之所以安全，是因为 **`range` 在进入循环前，对 `len(queue)` 做了一次“快照”（Snapshot）**。它就像给当前队列里的节点盖了个戳：“只有你们是这一层的，后面新来的都要排队等下一轮”。

所以，你可以放心地使用这种简洁写法，它既准确又高效！

# 题116: 填充每个节点的下一个右侧节点指针

本题依然是层序遍历，只不过在单层遍历的时候记录一下本层的头部节点，然后在遍历的时候让前一个节点指向本节点就可以了

```python
# @lc code=start
"""
# Definition for a Node.
class Node:
    def __init__(self, val: int = 0, left: 'Node' = None, right: 'Node' = None, next: 'Node' = None):
        self.val = val
        self.left = left
        self.right = right
        self.next = next
"""
# 本题依然是层序遍历，只不过在单层遍历的时候记录一下本层的头部节点，然后在遍历的时候让前一个节点指向本节点就可以了
# 注意: 本题要求的返回值类型是 Node（即修改后的根节点）
# 在数据结构中，root 确实是一个节点，但它是整棵树的入口。即使循环结束了，root 所指向的那个对象已经不是最初的那个“孤零零”的 root 了，它的子孙后代们的 next 指针全都已经被你在线下“偷偷”连好了。
# 我们在循环里做的 node.next = ...，实际上是在给珍珠之间加新的连线。当我们加完所有的 next 连线后，从 root 出发，判题系统就能顺着 left/right 往下走，顺着 next 往右走，从而检查整棵树的连接是否正确。
class Solution:
    def connect(self, root: 'Optional[Node]') -> 'Optional[Node]':
        if not root:
            return root # 如果 root 为空，queue 就是 [None], 注意: 此处返回的是节点, 不是list
        queue = collections.deque([root])
        # 注意: 本题不用新创建result数组, 题目要求的返回值类型是 Node（即修改后的根节点）, 看清楚题目要求

        while queue:
            level_size = len(queue)
            for i in range(level_size):
                # 弹出当前节点
                cur = queue.popleft()

                # 判断是否为当前层的最后一个节点
                if i < level_size - 1:
                    # 如果不是最后一个，它的 next 就在当前队列的最前面
                    cur.next = queue[0]
                else:
                    # 如果是最后一个，next 指向 None（初始默认也是 None）
                    cur.next = None
                # cur.next = queue[0] if i < level_size - 1 else None # [三元表达式Ternary operator]如果不是最后一个，指向 queue[0]，否则指向 None

                # 将cur的子节点加入队列queue
                if cur.left:
                    queue.append(cur.left)
                if cur.right:
                    queue.append(cur.right)
        return root
            
# @lc code=end
```

### 1. 为什么代码里不写 `result.append("#")`？

- **初始状态已默认为空**：根据题目描述，初始状态下所有 `next` 指针都被设置为 `NULL`。因此，如果代码没有显式修改最后一个节点的 `next` 指针，它自然就保持为 `None`，符合题目要求。

### 2. 代码逻辑解析

在你提供的 Python 模板代码中：

```

Q: 使用cur节点用python做法:

这种做法的核心思路是：当你从队列中取出当前节点 `node` 时，如果它不是当前层的最后一个节点（即 `i < level_size - 1`），那么它在队列中的“邻居”（即 `queue[0]`）就是它的 `next` 节点。

以下是基于这种思路的 **Python** 代码实现：

Python

```python
import collections

class Solution:
    def connect(self, root: 'Node') -> 'Node':
        if not root:
            return root
        
        queue = collections.deque([root])
        
        while queue:
            level_size = len(queue)
            
            for i in range(level_size):
                # 弹出当前节点
                node = queue.popleft()
                
                # 判断是否为当前层的最后一个节点
                if i < level_size - 1:
                    # 如果不是最后一个，它的 next 就在当前队列的最前面
                    node.next = queue[0]
                else:
                    # 如果是最后一个，next 指向 None（初始默认也是 None）
                    node.next = None
                
                # 将子节点加入队列
                if node.left:
                    queue.append(node.left)
                if node.right:
                    queue.append(node.right)
                    
        return root
```

### 两种方式的对比：

- **使用 `prev` 变量（截图中的做法）**：逻辑上是“处理当前节点时，让前一个节点指向我”。代码通过 `if prev: prev.next = node` 来实现。
- **使用 `queue[0]` 判断（你的提议）**：逻辑上是“处理当前节点时，我直接指向队列里的下一个”。这种写法通常看起来更简洁，因为它少维护一个变量，直接利用了队列的物理顺序。

**需要注意的地方：**

在 Python 的 `collections.deque` 中，访问 `queue[0]` 只是查看（peek）而不弹出，这与 `popleft()` 配合使用非常安全。

这是一个非常敏锐的直觉！在一般的循环中，如果直接访问 `queue[0]` 确实存在越界的风险。但在**这个特定的层序遍历代码结构**中，它是安全的。

我们来拆解一下为什么在这个 `for` 循环里 `queue[0]` 不为空：

### 1. 核心逻辑：`i < level_size - 1`

请注意这段代码中的判断条件：

Python

```
for i in range(level_size):
    node = queue.popleft() # 弹出当前节点
    
    if i < level_size - 1: # 这里的判断是关键！
        node.next = queue[0] 
```

- **当 `i < level_size - 1` 时**：说明当前节点后面**还有**同层的节点。因为我们这一层总共有 `level_size` 个节点，你才弹出了第 `i` 个，队列里肯定还剩下了这一层后续的节点。所以 `queue[0]` 此时指向的是同层的下一个节点，**绝对不为空**。
- **当 `i == level_size - 1` 时**：这是该层的最后一个节点。此时条件 `i < level_size - 1` 不成立，代码会跳过 `node.next = queue[0]`。所以即使此时队列里可能已经没有当前层的节点了（甚至队列可能是空的），我们也不会去访问 `queue[0]`。

------

### 2. 队列里的“新老交替”

你可能担心：*“如果我弹出了当前层最后一个节点，此时队列里已经加入了下一层的孩子节点，那 `queue[0]` 不就指向下一层的第一个了吗？”*

**没错，确实会指向下一层。** 但幸好我们的 `if i < level_size - 1` 挡住了这种情况：

| **迭代步骤** | **队列状态 (假设每层2个)** | **i 的值** | **i < level_size - 1?** | **操作**                  |
| ------------ | -------------------------- | ---------- | ----------------------- | ------------------------- |
| 处理第1个    | `[A, B]` (弹出A)           | 0          | **Yes** (0 < 1)         | `A.next = queue[0]` (即B) |
| 处理第2个    | `[B, C, D]` (弹出B)        | 1          | **No** (1 < 1 错误)     | 跳过，不访问 `queue[0]`   |

> **结论：** 只要 `i` 还没到最后一个，`queue[0]` 就永远是当前层的“右邻居”。一旦到了最后一个，我们就直接不去碰 `queue[0]`。

------

### 3. 更稳妥的写法（如果你担心的话）

如果你觉得访问 `queue[0]` 心理上还是觉得不踏实，Python 还有一种更“优雅”的写法，利用 `None`：

Python

```
for i in range(level_size):
    node = queue.popleft()
    # 如果不是最后一个，指向 queue[0]，否则指向 None
    node.next = queue[0] if i < level_size - 1 else None
    
    if node.left: queue.append(node.left)
    if node.right: queue.append(node.right)
```

这两者在逻辑上是完全等价的。

之所以在处理 $i=0$ 时队列是 `[2, 3]`，是因为**层序遍历（BFS）有一个核心规则：处理当前层之前，队列里已经完整地放好了这一层所有的节点。**

我们可以按照代码执行的顺序，从根节点（第一层）开始“慢动作回放”：

------

### 第一步：初始化

- **代码执行：** `queue = collections.deque([root])`
- **队列状态：** `[1]`

2. 进入 `for i in range(1)` 循环：

此时回到 `while` 循环的开头：

1. `level_size = len(queue)`，此时 **`level_size` 变成了 2**。
2. 进入 `for i in range(2)` 循环：

- **入队：** 把 2 的孩子 **4** 和 **5** 放进去。
- **此时队列状态：** `[3, 4, 5]`（3 在队首，4 和 5 在后面排队）。

#### 当 $i = 1$ 时（处理节点 3）：

- `node = queue.popleft()`，拿到节点 **3**。
- **此时队列剩下的内容：** `[4, 5]`。

- **赋值：** 跳过（3 的 `next` 保持为 `None`）。
- **入队：** 把 3 的孩子 **6** 和 **7** 放进去。

------

### 总结：为什么 $i=0$ 时队列是 `[2, 3]`？

是因为在处理**上一层**（第一层）结束时，代码已经通过 `queue.append` 把这一层的所有节点都“攒”齐了。

**你可以把 `for i in range(level_size)` 想象成一个“隔离区”：**

- 在进入这个 `for` 循环的一瞬间，这一层有多少人（`level_size`）就已经定死了。
- 即使你在循环里不断往队尾 `append` 新的孩子，这个 `for` 循环也只会处理原本在那里的那几个老节点。
- 只要还没轮到最后一个老节点，它的“右邻居”就一定还在队列的前端排队。

这种“先数人头再处理”的方法，是层序遍历最经典的地方。你现在觉得这个 `level_size` 带来的“确定性”好理解一些了吗？

- 循环走到 $i=0$

### 1. 英文名称

它最正式的英文名称是：

- **Ternary Conditional Operator**（三元条件运算符）
- 或者简称为 **Ternary Operator**

### 2. 语法结构

Python 的三元表达式结构比较特殊，读起来非常像自然语言（英语）：

> **`[结果为 True 时的值] if [判断条件] else [结果为 False 时的值]`**

对比其他语言（如 C++, Java, JavaScript）：

- **其他语言 (使用 `? :`):** `condition ? value_if_true : value_if_false`
- **Python:** `value_if_true if condition else value_if_false`

------

### 3. 为什么在这里用它？

在你的代码里：

Python

```
node.next = queue[0] if i < level_size - 1 else None
```

这行代码极其简洁地完成了两件事：

1. **逻辑判断**：检查当前节点是否是这一层的“老幺”（最后一个）。
2. **赋值**：如果是老幺，断开连接（`None`）；如果不是，指给队列里的下一个。

- **优点**：代码行数少，逻辑紧凑，可读性强（读起来就是：让 node.next 等于 queue[0]，如果 i 还没到头的话，否则等于 None）。
- **注意**：不要在三元表达式里写太复杂的逻辑，否则会变成“代码屎山”。但在这种简单的赋值场景下，它是**非常地道（Pythonic）**的写法。

Q: 最后返回return root的逻辑, 为啥不放在新创建的result数组里呢?  root不是只是一个节点吗

这是一个非常经典的问题，触及了**“值传递”与“引用传递”**，以及**“题目要求”**这两个核心概念。

### 1. 题目要求的本质：原地修改 (In-place)

在 LeetCode 116 题中，题目要求的返回值类型是 `Node`（即修改后的根节点）。

- **普通层序遍历（102题）**：目标是获取每一层的值，所以要定义 `res = []`。
- **本题（116题）**：目标是**改变树的物理结构**。你需要把树里成百上千个节点的 `next` 指针连起来。

你返回的虽然只是一个 `root` 节点，但由于每个节点都通过 `left`、`right` 和 `next` 指针相互连接，你返回的其实是**整棵已经连好了线的树**。

------

### 2. 为什么不用 `result` 数组？

如果你把节点放进 `result = [node1, node2, ...]` 数组里返回：

1. **格式不符**：判题系统期待得到一个根节点，然后通过这个根节点去验证所有 `next` 指针。如果你给它一个数组，它会报错。
2. **多此一举**：数组只是一个“容器”。这就好比你修理一辆自行车（树），你修好后把自行车（root）推给客户就行了，没必要把自行车的零件拆下来装进一个零件箱（result）里给客户。

------

### 3. 理解“root 不只是一个节点”

在数据结构中，`root` 确实是一个节点，但它是**整棵树的入口**。

> **形象比喻：**
>
> 想象一串珍珠项链。`root` 就是你拎在手里的那第一颗珍珠。
>
> 虽然你只拎着一颗，但因为珍珠之间都有线连着，你拎起第一颗，**整串项链（整棵树）就都被你提起来了。**

我们在循环里做的 `node.next = ...`，实际上是在**给珍珠之间加新的连线**。当我们加完所有的 `next` 连线后，从 `root` 出发，判题系统就能顺着 `left/right` 往下走，顺着 `next` 往右走，从而检查整棵树的连接是否正确。

------

### 4. 内存中的变化

当你执行 `node.next = queue[0]` 时，你修改的是**内存中那个真实对象的属性**。

Python

```
# 假设 root 指向内存地址 0x101
# node 也是指向 0x101 的一个引用
node = root 
node.next = some_other_node # 此时内存地址 0x101 处的对象的 next 属性变了
```

即使循环结束了，`root` 所指向的那个对象已经不是最初的那个“孤零零”的 `root` 了，它的子孙后代们的 `next` 指针全都已经被你在线下“偷偷”连好了。

- **116题（填充指针）**：我们要的是“成品”（修改了属性的树本身）。

Q: if not root:

这是一个非常棒的观察！在很多递归或迭代的二叉树题目中，你会发现代码的开头和结尾长得很像。

针对这道题，**结论是：确实可以合并，但保留它通常是为了代码的“鲁棒性（Robustness）”和“执行效率”。**

我们来对比一下两种写法：

### 1. 简洁版（不写 `if not root`）

如果你不写开头的判断，代码会直接运行到 `queue = collections.deque([root])`。

Python

```
def connect(self, root: 'Node') -> 'Node':
    # 如果 root 为空，queue 就是 [None]
    queue = collections.deque([root]) 
    
    while queue:
        level_size = len(queue)
        # 这里会出问题！
        for i in range(level_size):
            node = queue.popleft()
            if not node: # 你必须在这里额外判断 node 是否为空
                continue
            # ... 后续逻辑
    return root
```

**问题点：** * 如果 `root` 是空的，`queue` 里面会装一个 `None`。

- 在执行 `node.left` 或 `node.next` 时，程序会抛出 `AttributeError: 'NoneType' object has no height` 的错误。
- 所以，如果你不写开头的 `if not root`，你就必须在 `while` 循环内部写更多的判断来处理空节点。

------

### 2. 标准版（带 `if not root`） —— 也就是你看到的写法

Python

```
if not root:
    return root # 这里的 root 就是 None
```

**为什么要这样写？**

- **防御式编程（Guard Clause）**：这被称为“卫语句”。在程序的最开始就把异常情况（空树）拦截掉，后面的代码就可以放心地假设 `root` 是一定有内容的，不需要在循环里反复判断。
- **性能优化**：如果是一棵空树，程序直接在第一行就返回了。如果没有这个判断，程序还要去初始化队列、判断 `while` 条件、计算 `len`，白白浪费了资源。
- **语义清晰**：一眼就能看出这道题处理空树的逻辑。

------

### 3. “一行代码”合并技巧

如果你追求极致的简洁，在 Python 中可以利用 `and` 或 `or` 的短路特性，但**不推荐**在面试中这样写，因为可读性较差：

Python

```
def connect(self, root: 'Node') -> 'Node':
    if root:
        queue = collections.deque([root])
        # ... 剩余逻辑
    return root
```

这样写的话，如果 `root` 是空，直接跳过整个 `if` 块，最后 `return root`（即 `None`）。这和你说的逻辑是一致的。

------

### 总结

虽然 `return root` 在两种情况下返回的值都是 `None`，但开头的 `if not root` 像是一个**“保安”**，它确保了后续复杂的搬运工作（队列操作、指针连接）只在有“货物”（节点）的情况下才开展。

**这就好比：** 如果快递公司收到一个空包裹订单：

- **写法1（有if）**：保安看到订单是空的，直接拒收。
- **写法2（没if）**：快递员把空箱子装上车，开到目的地，打开发现是空的，再拉回来。

Q: prev = None 可以放在while queue之前吗

在 [117. 填充每个节点的下一个右侧节点指针 II](https://programmercarl.com/0102.二叉树的层序遍历.html#_117-填充每个节点的下一个右侧节点指针ii) 的 BFS 解法中，`prev = None` **不能**放在 `while queue` 之前。

### 为什么必须放在 `while` 循环内部？

这是因为 `prev` 变量的作用是**追踪当前行中上一个处理的节点**。

- **每一层都需要重置**：在处理每一层时，该层的第一个节点右侧没有其他节点（或者说它是新的开始），它的 `next` 指针不应该指向上一层的任何节点。
- **防止跨层连接**：如果把 `prev = None` 放在 `while` 之前，那么第一层的最后一个节点会被记录在 `prev` 中。 当程序开始处理第二层时，由于 `prev` 依然保留着第一层末尾节点的引用，代码中的 `if prev: prev.next = node` 会导致**第一层的末尾节点错误地指向了第二层的第一个节点**。

### 正确的逻辑位置

`prev` 必须在**每一层循环开始时**清空：

Python

```
while queue:
    level_size = len(queue)
    prev = None  # 正确：处理新的一层前，先清空前驱节点记录
    for i in range(level_size):
        node = queue.popleft()
        if prev:
            prev.next = node
        prev = node
        # ... 入队子节点逻辑
```

### 总结

- **放在 `while` 外面**：会导致树的结构变乱，每一层的结尾会连向下一层的开头。
- **放在 `while` 里面**：确保 `next` 指针只在**同层节点**之间建立连接。

# 题104: .二叉树的最大深度

```python
# @lc code=start
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right
# 使用迭代法的话，使用层序遍历是最为合适的，因为最大的深度就是二叉树的层数，和层序遍历的方式极其吻合。
# 在二叉树中，一层一层的来遍历二叉树，记录一下遍历的层数就是二叉树的深度
# 所以这道题的迭代法就是一道模板题，可以使用二叉树层序遍历的模板来解决的
class Solution:
    def maxDepth(self, root: Optional[TreeNode]) -> int:
        if not root:
            return 0
        
        queue = collections.deque([root])
        depth = 0 # 代表二叉树的层数
        
        while queue:# 代表queue存在, 因此此时depth不可能为0, 所有depth的创建需要在while外面
            depth += 1 # 我看到这一层存在，先把层数记下，再去处理这一层的所有节点
            for _ in range(len(queue)): # 取每层的节点, len(queue)代表每层的节点个数
                cur = queue.popleft()

                if cur.left:
                    queue.append(cur.left)
                if cur.right:
                    queue.append(cur.right)

        return depth
    
# @lc code=end
```



Q: depth+=1可以放在for循环之后么? 放前放后有区别吗

在 [104. 二叉树的最大深度](https://programmercarl.com/0102.二叉树的层序遍历.html#_104-二叉树的最大深度) 的 Python 层序遍历（BFS）做法中，`depth += 1` 的位置确实可以调整，但**放前放后是有逻辑区别的**。

### 1. 放在 `for` 循环之后可以吗？

**可以。** 只要它在 `while queue:` 循环内部，无论是 `for` 循环之前还是之后，其最终结果是一样的。因为 `while` 循环每执行一次就代表处理了完整的一层，只要进入了 `while` 循环，就说明这一层存在，深度就应该增加。

------

### 2. 放前与放后的区别

虽然结果相同，但它们在逻辑含义和初始值设定上略有细微差别：

| **depth += 1 位置**               | **逻辑含义**                                         | **推荐理由**                            |
| --------------------------------- | ---------------------------------------------------- | --------------------------------------- |
| **放在 `for` 之前**（如截图所示） | **“我看到这一层有人，先把层数记下，再去处理人。”**   | 比较直观。进入 `while` 就代表深度加 1。 |
| **放在 `for` 之后**               | **“我处理完了这一层的所有节点，层数才算真正增加。”** | 也是正确的逻辑。                        |

#### 注意事项：

- **不可放在 `for` 循环内部**：如果写成 `for _ in range(len(queue)): depth += 1`，那么深度会变成所有节点的总数，而不是层数。
- **初始值**：无论放前放后，只要 `root` 为空的情况已经在开头被 `if not root: return 0` 拦截，`depth` 初始值都应设为 **0**。

### 代码演示（放后的写法）

Python

```
while queue:
    for _ in range(len(queue)): # 处理当前层所有节点
        node = queue.popleft()
        if node.left: queue.append(node.left)
        if node.right: queue.append(node.right)
    depth += 1  # 处理完一层后，深度加 1
```

```python
# @lc code=start
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right

# 相对于 104.二叉树的最大深度 ，本题还也可以使用层序遍历的方式来解决，思路是一样的。
# 需要注意的是，只有当左右孩子都为空的时候，才说明遍历的最低点了。如果其中一个孩子为空则不是最低点
class Solution:
    def minDepth(self, root: Optional[TreeNode]) -> int:
        if not root:
            return 0
        
        queue = collections.deque([root])
        depth = 0

        while queue:
            depth += 1
            for _ in range(len(queue)):
                cur = queue.popleft()

                if not cur.left and not cur.right: # 只有当左右孩子都为空的时候，才说明遍历的最低点了。如果其中一个孩子为空则不是最低点
                    return depth
                
                if cur.left:
                    queue.append(cur.left)
                if cur.right:
                    queue.append(cur.right)
                    
        return depth
    
# @lc code=end
```
