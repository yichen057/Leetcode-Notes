# 题235: 二叉搜索树的最近公共祖先

```python
# @lc code=start
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, x):
#         self.val = x
#         self.left = None
#         self.right = None

class Solution:
    # 方法一: 递归法: 有左右遍历即可, 这里不涉及中
    def lowestCommonAncestor(self, root: 'TreeNode', p: 'TreeNode', q: 'TreeNode') -> 'TreeNode': # 返回值是最近公共祖先LCA
        # Base case
        if root is None:
            return None
        # 单层递归 Single layer recursion, 利用bts的有序性: 遍历的节点比p和q都大, LCA在左子树里, 向左遍历; 反之LCA在右子树, 向右遍历
        # 在 BST 中，从根节点到 LCA 的路径是唯一的：如果当前节点值比 $p, q$ 都大，LCA 必在左边。如果当前节点值比 $p, q$ 都小，LCA 必在右边。
        # 一旦不满足上述两条（即当前值在 $[p, q]$ 之间），当前节点就是 LCA，直接返回。具体原因见底部.
        # 这就意味着，如果你进入了左子树（left = traversal(cur.left, p, q)），并且它返回了一个非空值，说明它在左子树的深处已经触发了第 3 条规则（找到了终点）。根据递归的特性，这个结果会像回声一样直接传回来。
        # left
        if root.val > p.val and root.val > q.val: # 向左子树搜索找LCA, 因为递归函数有返回值, 所以得有变量接住
            left = self.lowestCommonAncestor(root.left, p, q) # left和right的数据类型是TreeNode
            if left is not None: # 搜索一条边的写法，遇到递归函数的返回值，如果不为空，立刻返回。
                return left # 左子树找到了LCA
        # right
        if root.val < p.val and root.val < q.val:
            right = self.lowestCommonAncestor(root.right, p, q)
            if right is not None:
                return right
        # 剩下的情况: 当前节点一定在p和q之间, 当p<root<q时, root一定是p和q的LCA
        # 因为p在root的左子树, q在root的右子树里. 如果继续向左遍历, 那么root将错过q; 如果继续向右遍历, 那么root将错过p. 所以只有root才能链接p和q, 此时root是p和q的LCA
        return root

    # 方法二: 迭代法
    def lowestCommonAncestor(self, root: 'TreeNode', p: 'TreeNode', q: 'TreeNode') -> 'TreeNode': # 返回值是最近公共祖先LCA
        while root:
            if root.val > p.val and root.val > q.val:
                root = root.left
            elif root.val < p.val and root.val < q.val:
                root = root.right
            else:
                return root
        return None # 在技术上是有必要写的，但从业务逻辑上几乎永远不会被执行到。
# @lc code=end
```

Q: leetcode235和236都是求最近公共祖先, 为啥236就得自下而上遍历, 而235就可以自上而下遍历呢? 二者有何区别和相同点呢

这是一个非常棒的观察！在二叉树的问题中，利用**数据结构的特性**往往能大幅简化逻辑。

以下是 [235. 二叉搜索树的最近公共祖先](https://www.bilibili.com/video/BV1Zt4y1F7ww) (BST) 与 **236. 普通二叉树的最近公共祖先** 的核心区别与联系：

------

### 1. 核心区别：利用“有序性”

两者最本质的区别在于**节点值的分布是否有规律**。

| **特性**     | **235. 二叉搜索树 (BST)**                              | **236. 普通二叉树**                                  |
| ------------ | ------------------------------------------------------ | ---------------------------------------------------- |
| **节点分布** | **有序**：左子树 < 根节点 < 右子树                     | **无序**：节点可能分布在任何位置                     |
| **寻找方向** | **自上而下**：可以通过比较值的大小直接判断去左还是去右 | **自下而上**：必须遍历到底部，再通过回溯把信息传上来 |
| **遍历方式** | 迭代或简单递归（类似二分查找）                         | 后序遍历（递归回溯）                                 |
| **关键判断** | 根节点值是否在 $[p, q]$ 之间                           | 左右子树是否分别发现了 $p$ 和 $q$                    |

------

### 2. 为什么 235 可以“自上而下”？

在 **BST** 中，如果你从根节点开始向下走：

- 如果当前节点的值 **大于** $p$ 和 $q$，说明 $p$ 和 $q$ 都在左子树，往左走。
- 如果当前节点的值 **小于** $p$ and $q$，说明 $p$ 和 $q$ 都在右子树，往右走。
- **一旦发现当前节点的值介于 $[p, q]$ 之间（或者等于其中之一）**，那么这个节点就是我们要找的“分叉点”，即最近公共祖先。

**结论：** 在 BST 中，第一个进入 $[p, q]$ 区间的节点就是 LCA，不需要回溯。

------

### 3. 为什么 236 必须“自下而上”？

在**普通二叉树**中，你只看当前节点的值，完全无法判断 $p$ 和 $q$ 是在左边还是右边。

- 你必须先深入到叶子节点，确认“我这一支里到底有没有 $p$ 或 $q$”。
- 然后通过**回溯**（即函数返回的过程），告诉父节点：“我找到了 $p$”或“我找到了 $q$”。
- 当某个父节点发现：**“我的左子树返还了信息，右子树也返还了信息”**，它才知道自己是那个 LCA。

**结论：** 普通二叉树没有值的大小指引，只能通过全量搜索后的“信息汇报”来定位。

------

### 4. 相同点

- **逻辑终点一致：** 两者都在寻找那个“分叉点”——即 $p$ 和 $q$ 第一次分属两个不同子树（或者其中一个是另一个的祖先）的位置。
- **边界情况：** 如果 $p$ 或 $q$ 本身就是祖先，两道题的逻辑都会在触达该节点时停止进一步深挖。

------

### 总结

- **235 (BST)：** 像是在**有导航**的高速上开车，根据路标（数值大小）直接选出口。
- **236 (普通树)：** 像是在**没有地图**的森林里找人，必须派人搜遍每一个角落，等所有人回来汇报位置。

你要是想深入理解 235 的代码实现细节，[这段视频讲解](https://www.bilibili.com/video/BV1Zt4y1F7ww) 里的迭代法部分非常清晰，它展示了为什么 BST 不需要回溯就能搞定。



Q: 236题里, 返回none是因为没有pq所以返回none, 235题返回none是因为同样的原因, 还是确实值为空呢?

在 [236. 二叉树的最近公共祖先](https://www.google.com/search?q=https://programmercarl.com/0236.%E4%BA%8C%E5%8F%89%E6%A0%93%E7%9A%84%E6%9C%80%E5%B0%8F%E5%85%AC%E5%85%B1%E7%A5%96%E5%85%88.html) 和 [235. 二叉搜索树的最近公共祖先](https://www.google.com/search?q=https://programmercarl.com/0235.%E4%BA%8C%E5%8F%89%E6%90%9C%E7%B4%A2%E6%A0%91%E7%9A%84%E6%9C%80%E5%B0%8F%E5%85%AC%E5%85%B1%E7%A5%96%E5%85%88.html) 中，返回 `None`（或 `NULL`）的逻辑确实有细微的差别，主要体现在**“搜索方式”**上。

### 1. 236题（普通二叉树）：返回 None 代表“没找到信号”

在 236 题中，递归是**自底向上**回溯的。

- **含义**：当你执行 `return None` 时，你是在告诉父节点：“报告！我这棵子树里**既没有 p 也没有 q**。”
- **逻辑**：父节点只有在 `left` 和 `right` 同时收到非空信号时，才会认定自己是祖先。如果两边都返回 `None`，它也会继续向上返回 `None`。

### 2. 235题（二叉搜索树）：返回 None 更多是“安全保底”

在 235 题中，利用了 BST 的有序性，递归是**从上向下**定向搜索的。

- **含义**：在 235 题的迭代法或递归法中，返回 `None` 通常出现在遍历到了叶子节点之外（`while root` 循环结束）。
- **逻辑差别**：
  - **BST 的特性**：因为 BST 有方向性（左小右大），我们是**带着目标区间去“找”**。
  - **题目保证**：题目明确说明了 `p` 和 `q` **均存在**于给定的二叉搜索树中。
  - **结论**：在 235 题中，只要树不为空，**逻辑上一定会在中途遇到最近公共祖先并提前 `return`**。最后的 `return None` 只是为了符合函数返回类型的要求（保底），在题目给定的条件下，代码其实很难运行到返回 `None` 那一步。

------

### 总结对比

| **题目**           | **场景** | **return None 的核心意义**                                   |
| ------------------ | -------- | ------------------------------------------------------------ |
| **236题 (普通树)** | 递归回溯 | **功能性返回**：明确告知父节点该路径下没有目标。             |
| **235题 (搜索树)** | 定向搜索 | **语法性返回**：逻辑上一定能找到，`None` 只是为了处理循环正常退出或树为空的极端情况。 |

### 为什么 235 题更简单？

正如 [235. 二叉搜索树的最近公共祖先](https://www.google.com/search?q=https://programmercarl.com/0235.%E4%BA%8C%E5%8F%89%E6%90%9C%E7%B4%A2%E6%A0%91%E4%B8%AD%E7%9A%84%E4%BC%97%E6%95%B0.html%23%E6%80%9D%E8%B7%AF) 所述，搜索树不需要遍历整棵树去“搜集信号”，它更像是在路口看路标：

- 如果 $p, q$ 都比当前节点小，往左拐；
- 如果 $p, q$ 都比当前节点大，往右拐；
- **一旦你发现自己在 $p$ 和 $q$ 之间**，你就已经站在终点了，直接 `return`。

Q: 为什么left非空, 就在左子树找到了LCA?

这是一个非常经典的回溯思维误区。我们需要明确：**`left` 非空并不一定代表找到了 LCA，它也可能仅仅是代表找到了 $p$ 或 $q$ 中的一个。**

在 [236. 二叉树的最近公共祖先](https://www.google.com/search?q=https://programmercarl.com/0236.%E4%BA%8C%E5%8F%89%E6%A0%93%E7%9A%84%E6%9C%80%E5%B0%8F%E5%85%AC%E5%85%B1%E7%A5%96%E5%85%88.html) 的递归逻辑中，`left` 的非空返回值具有 **“双重身份”**，它的含义取决于它从哪一层返回：

------

### 1. 身份一：它是“发现目标”的信号（底层返回）

当递归到叶子节点附近，发现了 $p$ 或 $q$ 时：

- `left` 返回的是 $p$（或 $q$）。
- 此时，对于父节点来说，`left` 非空仅仅意味着：**“左子树里有我们要找的其中一个目标。”**
- **此时它还不是 LCA**，它只是一个待定的信号。

### 2. 身份二：它是“已找到的 LCA”（中层返回）

当递归向上传递到某个节点，发现其左右子树刚好分别包含 $p$ 和 $q$ 时：

- 这个节点会 `return root`。
- 此时，这个 `root` 就是真正的 **LCA**。
- 这个 LCA 会作为非空值，被继续向上层传递。

------

### 3. 为什么代码里直接写 `return left`？

你看这段逻辑：

Python

```
if left is not None and right is not None:
    return root  # 只有两边都有，当前 root 才是 LCA

if left is not None: 
    return left  # 只有左边有，返回左边的“信号”
```

**关键点在这里：**

如果 `right` 是 `None`，说明右边啥也没找着。那么：

- 如果 `left` 是 $p$ 或 $q$，我们就把这个**目标信号**往上传。
- 如果 `left` 是已经找好的 **LCA**，我们也把这个**最终答案**往上传。

**结论：** 上层节点并不需要知道 `left` 到底是一个“目标”还是一个“结果”。它只需要知道“左边有东西”，然后继续向上接力。直到这个信号传到整棵树的根节点，它就是我们要的答案。

------

### 4. 形象理解：接力棒

你可以把 `left` 想象成一个**接力棒**：

- **第一棒**：某个节点在草丛里捡到了 $p$，它把 $p$ 当作接力棒传给爸爸。
- **第二棒**：爸爸发现另一边没东西，就把 $p$ 传给爷爷。
- **转折点**：爷爷发现左手接到了 $p$，右手接到了 $q$。爷爷很开心，把接力棒换成了**自己（LCA）**，再传给曾爷爷。
- **终点**：最终传到最高层手中的，无论是谁，都是我们要的 LCA。

### 总结

`left` 非空时，我们返回 `left`，不是因为它在这一层就是 LCA，而是因为它**携带了关键信息**（可能是目标，也可能是已经确定的 LCA）。我们通过 `return` 不断地把这个非空信息**推向更高层**。

这就解释了 [代码随想录](https://www.google.com/search?q=https://programmercarl.com/0236.%E4%BA%8C%E5%8F%89%E6%A0%93%E7%9A%84%E6%9C%80%E5%B0%8F%E5%85%AC%E5%85%B1%E7%A5%96%E5%85%88.html%23%E6%80%9D%E8%B7%AF) 中提到的：**“如果 left 为空，right 不为空，就返回 right，说明目标在右子树。”** 这个“目标”可以是单个节点，也可以是已经封装好的祖先节点。

你会发现这个逻辑在 [235. 二叉搜索树的最近公共祖先](https://www.google.com/search?q=https://programmercarl.com/0235.%E4%BA%8C%E5%8F%89%E6%90%9C%E7%B4%A2%E6%A0%91%E4%B8%AD%E7%9A%84%E4%BC%97%E6%95%B0.html) 中被极大地简化了，因为有了方向，我们连“接力”都不用了，直接在那儿撞见就行。你想对比一下这两者在“返回路径”上的区别吗？



Q:  迭代法最后一句return None有必要写吗?

在 Python 的逻辑实现中，最后一句 `return None` **在技术上是有必要的，但从业务逻辑上几乎永远不会被执行到。**

我们可以从以下两个层面来看待这行代码：

### 1. 语法完整性（为什么建议写）

- **防御性编程**：作为一个优秀的开发者，给函数提供一个“保底”的返回值是好习惯。如果因为某些不可预见的错误（比如传入的 `root` 本身就是 `None`，或者 `p`、`q` 不在树中），`while root` 循环会结束。如果没有最后一句，函数会默认返回 `None`。
- **代码规范**：显式地写出 `return None` 能够清晰地告诉阅读代码的人：“如果循环由于某种原因没找到结果，我会返回空。”这比让 Python 隐式返回要更具可读性。

### 2. 题目逻辑（为什么它碰不到）

根据 [235. 二叉搜索树的最近公共祖先](https://programmercarl.com/0235.二叉搜索树的最近公共祖先.html) 的题目说明：

> **p、q 为不同节点且均存在于给定的二叉搜索树中。**

这意味着：

- 只要树里有这两个点，根据二叉搜索树（BST）的性质，你**一定**会在 `else` 分支（即 `root` 的值处于 `p` 和 `q` 之间，或者等于其中之一时）撞见它们的最近公共祖先。
- 只要撞见了，就会执行 `return root` 并直接退出函数。
- 因此，`while` 循环正常结束（即 `root` 变为 `None`）的情况在符合题目要求的输入下是**不可能发生**的。

------

### 总结

| **情况**              | **是否需要写 return None**                                   |
| --------------------- | ------------------------------------------------------------ |
| **为了通过 LeetCode** | 不写也能过（因为 Python 默认返回 None）。                    |
| **为了代码严谨性**    | **建议写**。这符合“函数应当有明确出口”的编程原则。           |
| **为了逻辑理解**      | 明白它只是个“保底”，核心逻辑都在 `while` 循环里的 `return root`。 |

正如你在 [代码随想录的迭代法](https://programmercarl.com/0235.二叉搜索树的最近公共祖先.html#迭代法) 中看到的，作者保留了这行代码，就是为了保证函数结构的完整性。

**有趣的小观察**：

如果你把 `while root:` 改成 `while True:`，你甚至连最后一行都不用写，因为逻辑上它一定会从循环内部的 `return root` 飞出去。

# 题701: 二叉搜索树中的插入操作

```python
# @lc code=start
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right
# 方法一: 递归法
# 本题无需改变树的结构, 只要遇到空节点插入节点即可
# 无需遍历整棵树, BST是有方向的, 根据插入元素的值决定递归方向
# class Solution:
#     def insertIntoBST(self, root: Optional[TreeNode], val: int) -> Optional[TreeNode]:
#         # base case:
#         if root is None: # # 如果根节点为空，创建新节点作为根节点并返回. 当遇到叶子节点为空, 则在此处插入节点, 再将新插入的节点向上一层返回
#             node = TreeNode(val)
#             return node
        
#         # 向左遍历递归
#         if val < root.val:
#             root.left = self.insertIntoBST(root.left, val)

#         # 向右遍历递归
#         if val > root.val:
#             root.right = self.insertIntoBST(root.right,val)

#         return root
# 方法二: 迭代法
class Solution:
    def insertIntoBST(self, root: Optional[TreeNode], val: int) -> Optional[TreeNode]:
        if not root:
            return TreeNode(val)
        
        cur = root # 在二叉树的操作中，将 root 赋值给 cur（或其他辅助指针）, 是为了保护“树根”以便最后返回
        # 题目要求返回的是整棵树的根节点。
        # 在迭代过程中，我们需要不断地向下移动（例如 cur = cur.left）来寻找插入位置。
        # 如果直接使用 root 变量进行移动，当循环结束时，root 指针可能已经指向了树的底层节点甚至 None。
        # 后果： 你会“弄丢”整棵树的入口，无法返回题目要求的完整结果。
        while cur is not None:
            if val < cur.val:
                if not cur.left: # 如果此时父节点的左子树为空, 则在此插入新节点, 再向上返回给当前节点的左子树
                    cur.left = TreeNode(val)
                    return root
                else:
                    cur = cur.left
            elif val > cur.val:
                if not cur.right:
                    cur.right = TreeNode(val)
                    return root
                else:
                    cur = cur.right
                    
# 遍历与引用的分离
# 在算法实现中，通常会将变量的功能进行分离：
# root： 负责持有这棵树的句柄（Handle），作为常量或固定引用存在。
# cur (Current)： 负责执行具体的扫描任务，就像一个在树林里带路的“向导”，它会不断变换位置。
```

# 题450. Delete a node in a BST

```python
# @lc code=start
# Definition for a binary tree node.
# class TreeNode:
#     def __init__(self, val=0, left=None, right=None):
#         self.val = val
#         self.left = left
#         self.right = right
# 删除BST的节点: 删了节点后将改变树的结构, 且删后仍为BST
    # 递归法
class Solution:
    def deleteNode(self, root: Optional[TreeNode], key: int) -> Optional[TreeNode]:
        # Base Case:(此处的递归终止条件比较复杂): 本题不去遍历整棵树, 而是在遍历的过程中寻找要删的节点, 当找到要删的节点时就是终止条件, 所以删除节点的逻辑就是终止条件
        # 删除操作的逻辑之所以是终止条件，是因为 “删除”这个动作本身就是递归寻找过程的终点。一旦完成删除动作，当前子树的状态就确定了，剩下的就是一层层向上把这个结果传递回去。
#         在递归函数中，终止条件通常有两种：
# 1) 没找到：走到空节点了（`if root is None: return None`），说明任务失败，原样返回。
# 2) 找到了：既然找到了要删除的节点，那么本层递归的核心任务已经完成了。处理完删除逻辑后，不需要再向下递归，直接把处理后的结果（可能是新节点，也可能是 None）返回给上一层即可。
        # 情况一: "处理空树"及“没找到要删的节点
        if root is None:
            return root
        # 情况二: 找到了要删的节点. 说明当前的这个 `root` 变量所指向的节点内存地址，其值正好等于我们要找的 `key`, 所以，此时的 `root` 就是“目标靶子”
        if root.val == key:
            # 2.1 待删节点为叶子节点: 左右子节点为空, 未改变BST的结构
            if root.left is None and root.right is None:
                return None
            # 2.2 待删节点: 左子节点不空, 右子节点为空
            elif root.left and root.right is None: # 此处root.left可以省略不写
                return root.left
            # 2.3 待删节点: 左为空, 右不为空
            elif root.right and root.left is None: # 此处root.right可以省略不写
                return root.right
            # 2.4 待删节点: 左不为空, 右也不为空(核心!)
            else: # 利用BST左小右大的特性, root为待删节点
                cur = root.right # 先进入root的右子树, 为root的左子树找新家
                while cur.left is not None : # 这段循环的目标是找到右子树中数值最小的节点(右子树中最左侧的节点), 通过不断向左深入, 最终停下来的cur节点, 就是整棵右子树中值最小的节点
                    # 删掉 `root` 后：
                    # 1. 我们要把root的整个左子树搬走。
                    # 2. 搬到哪里呢？搬到root右子树的最左边节点的左孩子位置上。只要左边还有路, 就一直往左走
                    # 3. 因为右子树的最左节点是右边所有数里最小的，但它依然比左子树里的所有数都大。所以把左子树接在它的左边，依然符合 BST 的定义。
                    # 其实，找接班人有两种方案，效果是等价的：
                    # 方案 A（代码采用的）：找右子树的最小节点（Right Minimum）。
                    # 方案 B: 找左子树的最大节点（Left Maximum）。
                    cur = cur.left # cur由root的右子节点, 变为右子节点的左子节点. 这里的cur就是右子树里最小的节点
            cur.left = root.left # root的右子树里最小节点的左子树为原来root的左子树
            # 待删节点此时左子树已为空右子树不为空, 父节点指向要删节点的右孩子, 不经过待删节点
            root = root.right # root改指向原root的右子节点, 并赋值给root
            return root # 把原root的右子节点返回至上一层, 使得被删元素被跳过
        # 单层递归
        # 向左遍历
        if key < root.val:
            root.left = self.deleteNode(root.left, key)
        # 向右遍历
        if key > root.val:
            root.right = self.deleteNode(root.right, key)
        # 返回删除key后的新的BST的根节点
        return root
```

这一课由“代码随想录”的卡哥讲解，专门攻克 LeetCode 450. 删除二叉搜索树中的节点。这道题之所以被公认为二叉搜索树（BST）操作中最难的一道，是因为**删除节点可能会破坏整棵树的结构**，需要进行“挪移”和“重接”。

为了让你这个新手听得明白，我们将思路拆解为**五种情况**和**一个核心技巧**。

------

## 一、 核心思路：分类讨论

删除节点时，我们首先要根据目标值 `key` 在树中找到那个节点。一旦找到了，根据该节点的子树情况，处理方式完全不同：

| **情况** | **场景描述**                     | **处理方法**                                                 |
| -------- | -------------------------------- | ------------------------------------------------------------ |
| **1**    | **没找到**要删除的节点           | 直接返回 `NULL`，原路返回。                                  |
| **2**    | 目标是**叶子节点**（左右都为空） | 最简单，直接删掉，返回 `NULL` 给父节点。                     |
| **3**    | **左空右不空**                   | 类似于“继承”，让右孩子直接顶替自己的位置。                   |
| **4**    | **左不空右空**                   | 同理，让左孩子直接顶替自己的位置。                           |
| **5**    | **左右都不为空**                 | **最难的情况**：需要把左子树“寄托”到右子树的最左边，然后返回右孩子。 |

------

## 二、 难点攻克：左右都不为空怎么办？

这是视频中卡哥重点讲解的部分。当一个节点既有左子树又有右子树时，直接删掉它会导致两个子树“无家可归”。

**具体操作步骤：**

1. 找到该节点 **右子树的最左侧节点**（即右子树中数值最小的那个点）。
2. 把要删除节点的 **整棵左子树** 接到这个“最左侧节点”的左孩子位置上。
3. 删掉当前节点，返回 **右孩子** 作为新的根节点。

> **比喻理解：** 就像一个部门主管离职了，他带的一群员工（左子树）得找个新去处。最稳妥的办法是把这群员工交给接替者（右孩子）手下最基层、最能包容人的那位同事（右子树的最左节点）。

------

## 三、 代码逻辑拆解 (递归法)

卡哥的代码非常精妙，他利用了**递归函数的返回值**来完成节点的删除和重新连接。

C++

```
TreeNode* deleteNode(TreeNode* root, int key) {
    // 情况1：没找到
    if (root == nullptr) return root; 

    if (root->val == key) {
        // 情况2：叶子节点
        if (root->left == nullptr && root->right == nullptr) {
            delete root;
            return nullptr;
        }
        // 情况3：左空右不空
        else if (root->left == nullptr) {
            auto retNode = root->right;
            delete root;
            return retNode;
        }
        // 情况4：左不空右空
        else if (root->right == nullptr) {
            auto retNode = root->left;
            delete root;
            return retNode;
        }
        // 情况5：左右都不为空 (核心难点)
        else {
            TreeNode* cur = root->right; // 找右子树
            while (cur->left != nullptr) {
                cur = cur->left; // 找到右子树最左边的节点
            }
            cur->left = root->left; // 把被删节点的左子树挂上去
            TreeNode* tmp = root;   // 保存一下用来释放内存
            root = root->right;     // 让右孩子接替
            delete tmp;
            return root;
        }
    }

    // 递归寻找
    if (root->val > key) root->left = deleteNode(root->left, key);
    if (root->val < key) root->right = deleteNode(root->right, key);
    return root;
}
```

------

## 四、 给新手的理解建议

1. **返回值是关键**：`root->left = deleteNode(...)` 这一行的意思是，通过递归去子树里删节点，删完后返回新的子树根节点，再重新接回到当前的 `left` 上。
2. **BST 的特性**：始终记得左小右大。这也是为什么我们要把左子树接到右子树的“最左边”——因为左子树的所有值都比右子树小，挂在右子树最小值下面依然符合 BST 规则。
3. **内存释放**：在 C++ 代码中，手动 `delete` 节点是一个好习惯，虽然在 LeetCode 上不写通常也能过，但面试时这点很加分。

Q: **左不空右空**, 为啥return root.left就表示让左孩子直接顶替自己的位置? 这里的root指的是被删元素的父节点吗? 还是被删元素本身呢?

这是一个非常棒的问题，抓住了**递归删除**最核心的逻辑。很多新手在这里都会绕一下，我们直接拆解开来看：

### 1. 这里的 `root` 指的是谁？

在 `deleteNode(root, key)` 这个递归函数里，**`root` 指的是“当前正在被检查的节点”**。

当程序运行到 `if (root->val == key)` 里面的逻辑时，这个 `root` **正是那个“要被删掉的元素本身”**，而不是它的父节点。

------

### 2. 为什么 `return root->left` 就代表“顶替”？

要理解这一点，不能只看 `return` 这一行，必须看**调用它的上一层**。

在代码的最后，有两行关键的递归调用：

- `root->left = deleteNode(root->left, key);`
- `root->right = deleteNode(root->right, key);`

#### 举个形象的例子：

假设有一棵树，父节点是 **A**，A 的左孩子是 **B**（我们要删的目标），B 有一个左孩子 **B-left**。

1. **第一层递归**：我们在 A 节点，执行 `A->left = deleteNode(A->left, key)`。这意味着：**“我去把左儿子 B 处理一下，处理完后，谁该当我的新左儿子，你就给我返回谁。”**
2. **第二层递归**：进入 B 节点。发现 B 就是要删的。
3. **执行逻辑**：B 发现自己“左不空右空”。于是执行 `return root->left;`。
4. **发生顶替**：B 把自己的儿子 **B-left** “引荐”给了 A。
5. **回到第一层**：第一层的 `A->left` 接收到了这个返回值（B-left），直接把自己连向了 B-left。

**结果**：B 被跳过了（删除了），B-left 直接坐到了原来 B 的位置上，成了 A 的左孩子。这就是所谓的**顶替**。

------

### 3. 总结

- **`root`**：是被删节点本身。
- **`return root->left`**：是告诉父节点：“我已经消失了，以后我的左孩子就是你的新儿子了。”
- **赋值操作 (`root->left = ...`)**：才是真正完成“顶替”动作的那个“接线”过程。

> **比喻理解**：
>
> 就像一个老主管（父节点）让小王（被删节点）去办离职。小王走之前对老主管说：“我走了，这是我手下的小李（左孩子），以后他直接归你管。” 老主管点点头，把原来连向小王的电话线，直接插到了小李的工位上。

**这个“接线”的过程理解了吗？如果觉得抽象，我可以为你模拟一下“左右都不为空”时那个复杂的“左子树迁移”过程。**



Q: 情况5左右都不为空时, 这里的root和cur指代的是哪个节点? 

这个问题问得太关键了！理解递归代码时，最容易在大脑里把“变量名”和“具体的节点”混淆。

我们拿一个具体的数值例子来“肉眼跑一遍代码”，保证你瞬间通透。

### 场景模拟

假设我们要删除节点 **7**，此时当前的 `root` 就是 **7**。

它的左边挂着 **5**（及其子树），右边挂着 **9**。而 **9** 的左边还挂着一个 **8**。

- **当前 `root`**：节点 **7** (我们要删的对象)
- **`root->left`**：节点 **5**
- **`root->right`**：节点 **9**

------

### 第一步：`TreeNode* cur = root->right;`

- **谁是 `cur`？** 它是节点 **9**。
- **目的**：先进入右子树，准备为左子树找个“新家”。

### 第二步：`while (cur->left != nullptr) { cur = cur->left; }`

- **`cur` 变成了谁？** 节点 **9** 有左孩子 **8**，所以 `cur` 变成了节点 **8**。
- **目的**：我们要找右子树里**最小**的那个坑位。因为 BST 的特性，右子树里最小的节点（最左边）也比左子树里最大的节点要大，所以把左子树接在这里最安全。

### 第三步：`cur->left = root->left;`

- **动作**：把节点 **8** 的左手，牵起了节点 **5**。
- **结果**：现在节点 **5**（原来的左子树）已经成功“寄宿”到了右子树的最末端。

### 第四步：`root = root->right;`

- **动作**：这里的 `root` 本来指向 **7**，现在改指向了 **9**。
- **注意**：这只是在当前函数作用域里改了 `root` 的指向，真正的“接线”是靠最后的 `return root;`。

### 第五步：`return root;`

- **动作**：把节点 **9** 返回给上一层（也就是 **7** 的父节点）。
- **结果**：父节点原本连着 **7** 的线，现在咔哒一声，连到了 **9** 上。节点 **7** 彻底孤立，被 `delete tmp` 销毁。

------

## 总结：角色表

| **变量名**        | **在这个例子中代表谁**   | **它的最终命运**                                       |
| ----------------- | ------------------------ | ------------------------------------------------------ |
| **`root`**        | **被删节点 (7)**         | 被销毁，但在销毁前它把“王位”传给了右孩子。             |
| **`cur`**         | **右子树的最左节点 (8)** | 它是那个“接盘侠”，负责用自己的左手牵住被删掉的左子树。 |
| **`root->right`** | **继承人 (9)**           | 成功上位，变成父节点的新儿子。                         |
| **`root->left`**  | **被托付的左子树 (5)**   | 搬了家，从 7 的下面搬到了 8 的下面。                   |

### 💡 一个新手容易犯的小迷糊：

你可能会问：**“为什么不直接把 5 连在 9 下面，非要连在 8 下面？”**

因为节点 **9** 可能已经有左孩子（也就是 **8**）了！如果你直接 `9->left = 5`，那原来的 **8** 就丢了。所以我们必须一路向左找到最底层的 **8**，确保那里有个空位。

Q:  为啥场景5的最后一句return root是返回的7的父节点呢?

这是一个非常经典的点！其实，`return root` **返回的并不是父节点**，它返回的是**“由于 7 被删掉后，新上任的那个节点”**（在刚才的例子中是节点 9）。

真正“回到”父节点，是因为**递归函数执行结束了**。

我们可以把递归调用的过程想象成一串**对话**，这样你就能秒懂了：

------

### 场景对话模拟

假设父节点是 **6**，它的右孩子是 **7**（我们要删的节点）。

1. **父节点 (6) 发起请求**：

   父节点执行代码：`root->right = deleteNode(7, key);`

   **对话**：“喂，递归函数，请帮我把我的右儿子 **7** 删掉。删完之后，你告诉我，以后谁来当我的右儿子？”

2. **节点 (7) 内部处理**（也就是你问的情况 5）：

   - 节点 7 忙活了一阵，把自己的左子树挂到了右子树的最左边。

   - 节点 7 让 **9** 准备接班。

   - 节点 7 执行 `return root;`（此时这里的 `root` 已经指向了 **9**）。

     **对话**：“报告父节点，我已经处理好了！以后 **9** 就是你的新儿子了，收好它！”

3. **父节点 (6) 接收结果**：

   父节点那一层的代码 `root->right = deleteNode(...)` 终于等到了返回值。

   **动作**：父节点把自己的 `right` 指针，指向了传回来的那个 **9**。

------

### 为什么你会误以为返回的是父节点？

是因为**程序的控制权回到了父节点手中**。

- **`return root` 的作用**：是将“接班人”送出去。
- **谁接收了返回值**：是“上一层”递归函数。而对节点 7 来说，它的上一层恰好就是处理父节点 6 的那一层。

### 核心逻辑图示

Plaintext

```
父节点(6) 层:
   root->right = deleteNode(7, key)  <-- 1. 调用并等待结果
                      |
                      | (进入递归)
                      V
               节点(7) 层 (情况5):
                  ...处理挪动...
                  return 9;          <-- 2. 把 9 甩出去
                      |
                      | (跳回上一层)
                      V
父节点(6) 层:
   root->right = 9;                  <-- 3. 父节点接到 9，完成“接线”
```

### 总结

- **`return root`**：返回的是**新的子树根节点**（接班人）。
- **回到父节点**：是因为递归的“回溯”特性，当前任务完成了，程序自然跳回上一层继续执行。

**明白了吗？这个 `return` 其实是一个“引荐”动作。你现在能理解为什么删除操作一定要用 `root->left = ...` 这种赋值语句来接收返回值了吗？**



Q: 那为啥本题为啥不像701题一样, 要cur = root # 在二叉树的操作中，将 root 赋值给 cur（或其他辅助指针）, 是为了保护“树根”以便最后返回? 

这个问题问得非常精准！这反映了你对**递归**和**迭代**两种写法在底层逻辑上的深度思考。

简单来说：**701. 二叉搜索树中的插入操作** 如果采用**迭代法**，确实需要 `cur = root` 来保护树根；但本题（450题）采用的是**递归法**，它的保护机制和迭代法完全不同。

以下是详细对比，帮你理清这个逻辑：

------

### 1. 迭代法（通常在 701 题中使用）

在迭代法中，我们使用 `while` 循环不断向下寻找位置。

- **为什么要 `cur = root`？** 因为如果你直接用 `root = root->left` 去移动，等循环结束时，`root` 已经指向了树的最底层，你**找不到整棵树的头了**。
- **逻辑**：`cur` 像个探路兵去干活，`root` 像个总司令在终点站稳。

### 2. 递归法（卡哥在本题 450 题中的写法）

在递归法中，我们**不需要** `cur = root` 来保护，原因有二：

#### A. 每一层都有自己的 `root`

递归函数就像“俄罗斯套娃”。当你调用 `deleteNode(root->left, key)` 时，进入下一层函数后，那里的 `root` 其实是上一层的 `root->left`。

- **保护机制**：每一层递归都只负责处理自己那一小块地方。当你 `return` 的时候，它是返回给**上一层**的调用者。

#### B. 依靠“返回值”接线

看这行核心代码：

```
root->left = deleteNode(root->left, key);
```

- **真相**：这一层函数里的 `root`（也就是父节点）**从来没动过**！
- 它只是静静地等子递归返回一个结果，然后更新自己的 `left` 指针。
- 既然 `root` 本身没被重新赋值成别的节点，它就一直指向原来的位置，不需要额外保护。

------

### 3. 情况5 里的 `cur` 又是干嘛的？

你可能会问：“那 450 题情况 5 里不是也定义了一个 `cur` 吗？”

```
TreeNode* cur = root->right;
```

这里的 `cur` **不是为了保护树根**，而是为了**临时跑腿**：

- 此时我们要删掉当前的 `root`。
- 我们需要一个临时的指针 `cur` 跑到右子树的最左边，去把左子树“接”过去。
- 这个 `cur` 只在情况 5 这一小块逻辑里有用，它处理的是**局部重组**，不影响整体递归寻找的过程。

------

### 总结对比

| **比较维度**   | **迭代法 (常用 701)**             | **递归法 (本题 450)**                  |
| -------------- | --------------------------------- | -------------------------------------- |
| **指针移动**   | 必须移动指针来寻找节点            | 靠进入下一层函数来寻找节点             |
| **如何找回根** | 提前存一个 `cur` 指向初始 `root`  | 依靠递归函数层层返回，回到最初的调用点 |
| **变量含义**   | `root` 往往被当作起始位置保护起来 | `root` 是当前这一层子树的“临时负责人”  |

> **一句话理解**：
>
> 迭代法像是在**走迷宫**，你得在门口拉根绳子（`cur = root`）才能找回来；
>
> 递归法像是在**派下属办事**，你坐在办公室里不动，下属办完事把结果报送（`return`）给你就行，你不需要离开办公室。

**所以，在 450 题的递归写法里，直接操作 `root` 是安全的，因为每一层的 `root` 都是独立的“负责人”。你现在能分清“跑腿的 cur”和“负责人的 root”了吗？**



Q: if root is None:

​            return root

这里的if条件说的是没找到要删的节点, 还是二叉树为空?



这个问题问得非常到位，答案是：**两者皆是**。

在递归函数中，这行代码承载了两个层面的含义，取决于它是**第一次执行**还是在**递归过程中执行**。

------

### 1. 第一层面：处理空树（初始状态）

如果你调用 `deleteNode(None, 7)`，即传入的整棵树本来就是空的。

- **逻辑**：代码运行到这里，发现 `root` 是 `None`，直接返回 `None`。
- **含义**：告诉程序“树是空的，没啥好删的”。

### 2. 第二层面：没找到目标节点（递归过程中）

这是新手最容易困惑的地方。当你在树中寻找 `key` 时，递归会不断向左或向右子树深入。

- **逻辑**：假设你要找 `7`，但你一路找到了叶子节点都没找到，递归会继续向叶子节点的子节点（即 `None`）搜索。
- **含义**：当你进到了 `None` 这一层，说明**你已经走过了头，越过了叶子节点，却依然没碰到 key**。
- **返回过程**：此时 `return None` 会返回给上一层（叶子节点）。叶子节点执行 `root->left = deleteNode(...)` 接收到的还是 `None`。这相当于对原树**不作任何修改**。

------

### 3. 为什么代码可以写得这么简洁？

你会发现“处理空树”和“没找到节点”居然共用了一行代码。这是因为递归的**终止条件**（Base Case）通常都是处理最底层的情况。

我们可以把这个逻辑翻译成大白话：

> **递归函数说**：“只要我拿到的这个节点是空的（不管是因为树本来就空，还是因为我找过头了），我就直接把这个‘空’原样返回给我的上级。这意味着这层搜索到此为止，没有发生任何删除动作。”

------

### 总结

- **如果是刚进函数就触发了**：说明**二叉树为空**。
- **如果是递归了好几层才触发**：说明在当前的搜索路径上**没找到要删的节点**。

Q:  \# 情况二: 找到了要删的节点

​        if root.val == key:

这里的root是要删的节点吗



是的，在这一行代码执行时，**`root` 正是要被删除的那个节点**。

为了让你更直观地理解，我们可以从递归的视角来看待这个过程：

### 1. 为什么它是要删的节点？

在 [450. 删除二叉搜索树中的节点](https://programmercarl.com/0450.删除二叉搜索树中的节点.html) 的递归逻辑中：

- 程序通过 `if root.val > key` 或 `if root.val < key` 不断向子树深入。
- 一旦满足 `if root.val == key`，说明当前的这个 `root` 变量所指向的节点内存地址，其值正好等于我们要找的 `key`。
- **所以，此时的 `root` 就是“目标靶子”。**

### 2. 找到它之后发生了什么？

一旦确定 `root` 是要删的节点，代码会根据它的子树情况进行“外科手术”式的调整。你可以结合这张逻辑图来理解：

根据你提供的页面代码，处理逻辑如下：

- **无子节点（叶子）：** 直接返回 `None`（相当于告诉父节点：把你指向我的那个指针设为空）。
- **只有一个孩子：** 返回那个孩子（相当于告诉父节点：我走了，让我儿子顶替我的位置）。
- **有两个孩子（最复杂）：** 1.  找到右子树最左边的节点（也就是右子树里最小的）。
  2. 把被删除节点（当前的 `root`）的左子树接到那个最左节点的左边。
  3. 最后返回 `root.right` 顶替原位置。

### 3. 一个容易混淆的点

这里的 `root` 是一个**形式参数**。在递归调用中：

- 第一层调用的 `root` 是整棵树的根。
- 第二层调用的 `root` 可能是第一层的 `root.left`。
- **结论：** 在 `if root.val == key` 这一层级里，`root` 代表的是“当前正在处理的这棵子树的根节点”，而恰好这棵子树的根就是我们要删的节点。

------

**这个“第五种情况”（左右孩子都不为空）的节点移动逻辑，需要我帮你画个简单的示意图演示一下它是怎么“接过去”的吗？**



Q:  while cur.left is not None:

​                    cur = cur.left

这里用while循环, 是想找右子树最底部的左叶子节点吗? 

没错，你的理解已经非常接近了，但有一个小细节需要精确一下：

这段循环的目标是找到**右子树中数值最小的节点**。

在二叉搜索树（BST）中，最小的值一定位于一棵树“最左下角”的节点。所以，这段代码的具体意图是：

### 1. 寻找“右子树的最左节点”

当你准备删除节点 `root`，且它既有左孩子又有右孩子时（即“第五种情况”）：

- 代码先进入右子树：`cur = root.right`。
- 然后通过 `while cur.left is not None` 不断向左深入。
- **终点：** 最终停下的那个 `cur` 节点，是整棵右子树中**值最小**的节点。

### 2. 为什么要找它？

根据 [代码随想录](https://programmercarl.com/0450.删除二叉搜索树中的节点.html) 提供的逻辑，为了保持 BST 的特性（左小右大），删除 `root` 后：

1. 我们要把 `root` 的整个**左子树**搬走。
2. 搬到哪里呢？搬到 `root` **右子树的最左边节点的左孩子**位置上。
3. 因为右子树的最左节点是右边所有数里最小的，但它依然比左子树里的所有数都大。所以把左子树接在它的左边，依然符合 BST 的定义。

### 3. 注意：它不一定是“叶子节点”

虽然你提到了“左叶子节点”，但在实际情况中：

- 这个 `cur` 节点在循环结束时，它的 `left` 一定是 `None`。
- 但它的 **`right` 可能还有值**（即它可能是一个只有右孩子的度为 1 的节点）。
- 所以准确的说法是：**右子树中最左侧的节点**（Successor 的一种变形处理方式）。

------

### 总结代码行为

Python

```
cur = root.right # 1. 先迈向右子树
while cur.left is not None: # 2. 只要左边还有路，就一直往左走
    cur = cur.left # 3. 这里的 cur 就是右子树里最“小”的那个英雄
```

**这一步操作完后，紧接着的那句 `cur.left = root.left` 就是把孤儿（左子树）接回新家的关键，这一块逻辑你现在清晰了吗？**



Q :为啥要找最小的节点呢

在删除一个**既有左孩子又有右孩子**的节点时，我们面临一个难题：为了保持二叉搜索树（BST）的特性（左小右大），必须找一个节点来顶替被删除节点的位置。

之所以要找**右子树中最小的节点**（或者左子树中最大的节点），是因为它在数学逻辑和树结构上最稳妥。

------

### 1. 它是最完美的“接班人”

想象一下，要把一个国家的“国王”（被删节点）换掉，为了保证社会稳定（BST特性），新国王必须满足：

- 比所有“左子民”（左子树）都大。
- 比所有“右子民”（右子树）都小。

**右子树中最小的节点**（也就是右子树最左下的节点）刚好满足这个条件：

- 因为它来自右子树，所以它天然比左子树的所有节点都大。
- 因为它已经是右子树里最小的了，所以它一定比右子树剩下的所有节点都小。

### 2. 结构变动最小

在[代码随想录](https://programmercarl.com/0450.删除二叉搜索树中的节点.html)提供的这个解法中，逻辑其实非常巧妙：

1. 它并不是真的把这个“最小节点”挪上去，而是**把整个左子树“寄养”在这个最小节点的左边**。
2. 因为这个节点是右子树里最小的，它的左孩子位置**一定是空的**（如果是满的，它就不是最小了）。
3. 把左子树接过去后，整个右子树依然维持着“右边比左边大”的结构，此时直接让 `root.right` 整体上位即可。

### 3. 两种选择（对称性）

其实，找接班人有两种方案，效果是等价的：

- **方案 A（代码采用的）：** 找右子树的最小节点（Right Minimum）。
- **方案 B：** 找左子树的最大节点（Left Maximum）。

| **找谁**       | **位置**       | **逻辑**             |
| -------------- | -------------- | -------------------- |
| **右子树最小** | 右子树最左下角 | 把左子树接在它的左边 |
| **左子树最大** | 左子树最右下角 | 把右子树接在它的右边 |

------

### 总结

之所以找它，是因为在 BST 中，这个节点是**数值上最接近被删节点**的“邻居”。只有找邻居来接替位置，才不会破坏整棵树“左小右大”的严密秩序。

Q: 为什么删除节点的逻辑就是终止条件

这是一个非常敏锐的观察！在[代码随想录](https://programmercarl.com/0450.删除二叉搜索树中的节点.html)的递归实现中，`if (root.val == key)` 确实既是**“找到目标”**的时刻，也是该层递归的**“终止时刻”**。

我们可以从以下两个维度来拆解这个逻辑：

### 1. 递归的本质：任务完成即返回

在递归函数中，终止条件通常有两种：

- **没找到：** 走到空节点了（`if root is None: return None`），说明任务失败，原样返回。
- **找到了：** 既然找到了要删除的节点，那么**本层递归的核心任务已经完成了**。处理完删除逻辑后，不需要再向下递归，直接把处理后的结果（可能是新节点，也可能是 None）返回给上一层即可。

### 2. 为什么不再继续向下递归？

如果 `root.val == key` 成立，意味着当前的 `root` 就是要被干掉的对象。

- 你已经通过逻辑（比如找右子树最左节点）把树的结构重新接好了。
- 此时你返回的是**接好后的新子树根节点**。
- 对于当前这一层调用来说，它已经完成了“删除并重构”的所有工作，**它没有必要、也不应该**再去对比 `key` 和子树的值了。

------

### 3. 逻辑对比：搜索 vs. 删除

为了方便理解，你可以对比一下“搜索”和“删除”的逻辑差异：

| **操作类型**      | **找到目标后干什么？**        | **为什么是终止？**                 |
| ----------------- | ----------------------------- | ---------------------------------- |
| **搜索 (Search)** | 直接 `return root`            | 目标已达成，无需再找。             |
| **插入 (Insert)** | 在空位 `return TreeNode(val)` | 找到位置并安家了，任务结束。       |
| **删除 (Delete)** | **重构子树并返回新根**        | 节点已移除，结构已调整，任务结束。 |

### 4. 代码结构上的“错觉”

你可能会觉得它看起来不像 `if root is None` 那样典型的终止条件，是因为它后面还有逻辑：

Python

```
if root.val == key:
    # ... 复杂的删除与重构逻辑 ...
    return root.right # 或者其他返回值。一旦 return，该层递归就结束了。

# 下面这些是“没找到”才执行的
if root.val > key: root.left = self.deleteNode(root.left, key)
...
```

在程序逻辑中，只要执行了 `return`，这一层函数就彻底关闭了。所以 `if root.val == key` 后面的 `return` 语句，在效果上等同于“找到了，处理完，收工”。

------

**总结一下：**

删除操作的逻辑之所以是终止条件，是因为 **“删除”这个动作本身就是递归寻找过程的终点**。一旦完成删除动作，当前子树的状态就确定了，剩下的就是一层层向上把这个结果传递回去。
