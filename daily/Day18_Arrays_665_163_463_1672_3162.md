```python
        for i in range(len(nums)-1):

            if nums[i] > nums[i+1]:
                count += 1 # 发现断层, 记录一次修改
                if count > 1:
                    return False
                
                # 第二层：这是【决策比较】: 只有在“发现断层”后，也就是前提条件是nums[i]>nums[i+1]我们才开启这个逻辑，决定如何修改

# 2. 决策时刻**：我们要改掉 `4` 或者 `2`。

# 4. 拿 `nums[i+1]`(2) 和 `nums[i-1]`(1) 比：结果是 `2 >= 1`。
#    - **结论**：哪怕我把 `4` 削矮成 `2`，它依然比前面的 `1` 大。**最优决策**：削矮 `4`。

                    pass # 方案 A：削矮 nums[i]，对未来无影响，所以 pass. 想象中修改了 nums[i] = nums[i + 1]，对下一次 i+1 和 i+2 的比较没有副作用，所以不用写

                else:
                    nums[i+1] = nums[i] # # 方案 B：垫高 nums[i+1]，影响未来. 必须调大 nums[i + 1]，这会影响下一轮循环. 变大的是“未来”，必须更新，否则下一轮拿旧的 `2` 去比 `3` 就漏判了

        return True
```

### **1. 为什么属于 Arrays (数组)？**

- **基础结构**：题目直接操作 `nums` 数组，核心在于**索引（Index）管理**。
- **空间局部性**：你需要通过 `nums[i]`、`nums[i+1]` 甚至 `nums[i-1]` 的相互关系来做判断，这是典型的数组线性扫描题目。

### **2. 为什么属于 Greedy (贪心算法)？**

这道题是**贪心思想**的经典应用。

- **局部最优解**：当你发现 `nums[i] > nums[i+1]` 时，你必须立刻做出一个决定（是把大的调小，还是把小的调大）。
- **决策原则**：
  - **贪心策略**：我们希望修改后的数值给后面留下的“负担”越小越好。
  - **优先选择**：优先尝试把 `nums[i]` 变小（调成 `nums[i+1]`），因为这样能让前面的数尽量小，从而更容易满足后续的非递减条件。
  - **被迫选择**：只有当“把大的调小”会导致它比“前前个元素”还小时，我们才不得不把 `nums[i+1]` 调大。
- **全局最优**：通过每一步（局部）都尽可能保留更大的增长空间，最终达到判断整个数组能否通过 1 次修改完成的目标。

这是一个非常经典且容易出错的题目。你直觉提到的“遍历并比较前后元素”是核心，但处理 **“修改一个元素”** 时需要非常细致的逻辑。

------

### 1. 核心矛盾：发现“逆序对”怎么办？

当你发现 `nums[i] > nums[i + 1]` 时（即前大后小，破坏了非递减规则），你有两种修改选择：

1. **把前面的调小**：`nums[i] = nums[i + 1]`
2. **把后面的调大**：`nums[i + 1] = nums[i]`

**关键问题：** 到底改哪一个？如果改错了，可能会导致后面又出现不符合规则的情况。

------

### 2. 贪心策略判断逻辑

我们遍历数组，每当遇到 `nums[i] > nums[i + 1]` 时：

- **计数**：首先记录修改次数 `count += 1`。如果 `count > 1`，直接返回 `False`。
- **决策修改谁**：我们要观察 `nums[i - 1]`（前前个元素）来决定。
  - **情况 A**：如果是第一个元素就错了（$i=0$），或者 `nums[i + 1] >= nums[i - 1]`。
    - **做法**：牺牲 `nums[i]`，让它变小。因为把大的变小永远比把小的变大更安全（给后面留出更多空间）。
    - **代码体现**：不需要真的修改值，或者模拟 `nums[i] = nums[i + 1]`。
  - **情况 B**：如果 `nums[i + 1] < nums[i - 1]`。
    - **做法**：只能把 `nums[i + 1]` 调大，让它等于 `nums[i]`。
    - **代码体现**：必须执行 `nums[i + 1] = nums[i]`，因为这会影响下一次遍历的比较。

------

### 3. 代码参考实现

在 CodePath 的 [Problem 4: Non-decreasing Array](https://www.google.com/search?q=https://courses.codepath.org/courses/tip103/unit/1%23problem-4-non-decreasing-array) 中，你可以这样实现：

```python
def non_decreasing(nums):
    count = 0
    for i in range(len(nums) - 1):
        if nums[i] > nums[i + 1]:
            count += 1
            if count > 1:
                return False
            
            # 决定如何修改
            # 如果是首对逆序，或者修改 nums[i] 就能解决问题
            if i == 0 or nums[i + 1] >= nums[i - 1]:
                # 想象中修改了 nums[i] = nums[i + 1]，
                # 对下一次 i+1 和 i+2 的比较没有副作用，所以不用写
                pass
            else:
                # 必须调大 nums[i + 1]，这会影响下一轮循环
                nums[i + 1] = nums[i]
                
    return True
```

### 4. 为什么要“修改”数组？

如果不修改数组，在 **情况 B** 下（例如 `[3, 4, 2, 3]`）：

1. 走到 `i=1` 时，发现 `4 > 2`。
2. 如果你不把 `2` 改成 `4`，下一轮 `i=2` 时，你会用 `2` 去和 `3` 比较，结果是 `2 < 3`（看起来是对的）。
3. 但实际上数组变成了 `[3, 4, 4, 3]` 才是正确的修改，由于你没改值，程序会漏掉后面 `4 > 3` 这个错误。

### 💡 总结思路：

- **找错**：只允许出现一次 `nums[i] > nums[i + 1]`。

- **改错**：优先把大的调小；如果调小会导致比前面还小，就只能把小的调大。

  

Q: 展开讲解贪心策略, 我不明白第一个元素就错了是什么意思? 如果nums[i + 1] >= nums[i - 1]。为什么不是和nums[i]比较呢? 不用写代码nums[i]=nums[i+1]? 为什么把大的变小永远比把小的变大更安全? 

这个问题问得非常精准，触及了**贪心算法**在处理数组逻辑时的核心：**“如何修改才能为未来留出最大的容错空间？”**

为了让你彻底理解，我们把这个数组想象成一个“正在逐级上升的台阶”。

### 1. 为什么“把大的变小”比“把小的变大”更安全？

非递减数组的要求是 `nums[i] <= nums[i+1]`。

- **把小的变大**：意味着你强行抬高了当前台阶的高度。这会导致**后续的台阶也必须跟着变高**，否则后面很容易再次出现“逆序对”。
- **把大的变小**：意味着你降低了当前台阶的高度。这为后面的台阶留出了**更多的下降空间**（或者说，后面的数更容易比现在的数大）。

**结论**：在满足条件的前提下，数值越小，对后面的限制就越小。

------

### 2. “第一个元素就错了”是什么意思？

指的是当 `i = 0` 时发现 `nums[0] > nums[1]`。

- **例子**：`nums = [4, 2, 3]`
- 此时 `i=0`，`nums[0]=4`, `nums[1]=2`。
- 因为 `4` 面前没有更前面的元素了（没有 `nums[-1]`），所以我们**无脑把 4 改小**成 `2` 即可。
- 修改后变成 `[2, 2, 3]`。这显然比把 `2` 改大成 `4`（变成 `[4, 4, 3]`，后面又错了）要好得多。

------

### 3. 为什么是和 `nums[i-1]` 比较，而不是 `nums[i]`？

当你发现 `nums[i] > nums[i+1]`（比如 `4 > 2`）时，`nums[i]` 本身就是一个“坏人”。如果你拿 `nums[i+1]` 去跟“坏人”比，那是没有意义的。

我们需要参考的是**已经确定是正确的前一个台阶 `nums[i-1]`**。

- **情况 A (`nums[i+1] >= nums[i-1]`)**：
  - **例子**：`[1, 4, 2]`。这里 `i=1`，`nums[i-1]=1`, `nums[i]=4`, `nums[i+1]=2`。
  - 我们发现 `2` 虽然比 `4` 小，但它比前一个数 `1` 大。
  - 所以我们可以放心大胆地把 `4` 降级成 `2`。数组变成 `[1, 2, 2]`，完美！
- **情况 B (`nums[i+1] < nums[i-1]`)**：
  - **例子**：`[3, 4, 2]`。这里 `i=1`，`nums[i-1]=3`, `nums[i]=4`, `nums[i+1]=2`。
  - 此时 `2` 太小了，它甚至比前面的 `3` 还要小！如果你把 `4` 降级成 `2`，数组变成 `[3, 2, 2]`，依然是不合法的。
  - **没办法**，只能牺牲 `nums[i+1]`，把它强行拔高成 `4`。数组变成 `[3, 4, 4]`。

------

### 4. 为什么情况 A 不用写 `nums[i] = nums[i+1]`？

这是因为 Python 循环的特性。

- 在 `for i in range(len(nums) - 1)` 中，当你在索引 `i` 处发现问题并“假装”修改了 `nums[i]` 后，循环下一步会移动到 `i+1`。
- 下一轮比较的是 `nums[i+1]` 和 `nums[i+2]`。
- 既然你在情况 A 中是把前面的 `nums[i]` 调小了，它**不会影响到 `nums[i+1]` 之后任何数字的比较**。
- **反之**，在情况 B 中，你把 `nums[i+1]` 调大了，这直接影响了下一轮 `nums[i+1]` 与 `nums[i+2]` 的比较，所以**必须真写代码去修改值**，否则下一轮循环拿到的还是那个旧的小值。

------

### 💡 总结一个顺口溜：

> 遇到逆序先看前，
>
> 如果后数大于前（前前个），
>
> 压低当前保平安（不需改值）；
>
> 如果后数小于前（前前个），
>
> 只能拔高下一关（必须改值）。

这就是 [Non-decreasing Array](https://www.google.com/search?q=https://courses.codepath.org/courses/tip103/unit/1%23problem-4-non-decreasing-array) 这道题最精妙的地方。



Q: 如果是相邻元素比较, 那为啥贪心决策又要比较前前的元素? 是每一轮都要这样比较吗

我非常理解你的困惑，这正是这道题最“绕”的地方：**既然我们要的是相邻非递减，为什么眼光要看到“前前个”元素？**

原因很简单：当你发现 `nums[i] > nums[i+1]` 这个“断层”时，你不仅要修补当前的断层，还得保证**修补后的结果不能破坏之前已经修好的路**。

------

### 1. 为什么每一轮都要看“前前个”？

并不是每一轮。只有当你**发现错误**（即 `nums[i] > nums[i+1]`）时，你才需要看 `nums[i-1]`。

- **平时（没发现错）**：你只管手拉手往后走。
- **出事了（发现错）**：你现在必须改掉 `nums[i]` 或 `nums[i+1]` 中的一个。为了决定改哪一个，你必须回头看一眼 `nums[i-1]`，确保你的改动不会导致 `nums[i-1]` 和新改的数之间又出现了新的“断层”。

------

### 2. 举个“修路”的例子

想象你正在修一段上升的斜坡。每一块砖都要比前一块砖高。

现在你发现第 `i` 块砖太高了，第 `i+1` 块砖太低了（`nums[i] > nums[i+1]`）。你有两个修理方案：

#### 方案 A：把第 `i` 块砖削矮

你要把它削到和 `i+1` 一样高。

- **风险**：如果把它削矮了，它会不会变得比第 `i-1` 块砖还要矮？
- **判断依据**：看 `nums[i-1]`。如果 `nums[i+1] >= nums[i-1]`，说明削矮了也没事，它依然比前面的高。**这就是“安全”的改法。**

#### 方案 B：把第 `i+1` 块砖垫高

你要把它垫到和 `i` 一样高。

- **代价**：第 `i+1` 块砖变高了，意味着**后面所有的砖**都必须从这个更高的高度开始往上盖。
- **什么时候被迫用它**：只有当“方案 A”行不通（即削矮了会比 `i-1` 还矮）时，你才不得不动用这个代价更大的方案。

------

### 3. 具体的对比流程图

假设数组：`[3, 4, 2]`，此时 `i=1`（数字 4），`i+1`（数字 2），`i-1`（数字 3）。

1. **发现错误**：`4 > 2`。
2. **看一眼前前个**：`nums[i-1]` 是 `3`。
3. **做决策**：
   - 如果我想削矮 `4` 变成 `2` $\rightarrow$ 数组变成 `[3, 2, 2]` $\rightarrow$ **不合格！**（`3 > 2` 了）。
   - **结论**：削矮方案失败，必须垫高后数。执行 `nums[i+1] = nums[i]`，数组变成 `[3, 4, 4]`。

------

### 💡 核心总结 (重要!)

- **平时**：只比相邻。
- **遇错时**：回头看 `i-1`。
- **目的**：确保修改 `i` 或 `i+1` 后，`nums[i-1] <= nums[i] <= nums[i+1]` 这个链条依然完整。

**所以，并不是每一轮都要比较三个数。只有在“消耗那唯一一次修改机会”的瞬间，我们需要 `nums[i-1]` 提供决策参考。**

Q: 为什么情况 A 不用写 `nums[i] = nums[i+1]`？如果不写nums[i]=nums[i+1], 那每轮比较的不是隔一个元素比较吗? 那不会比较nums[i]和nums[i+2]?

这是一个非常棒的观察！你的疑问点在于：**如果不改值，下一轮循环比较时，`nums[i+1]` 是在跟谁比？**

让我们拆解一下循环的步进过程，你会发现 **“不写赋值语句”** 并不是 **“不比较”**，而是因为 **“前一个台阶已经变矮了，不会挡住后一个台阶”**。

------

### 1. 核心误区澄清：循环永远是“手拉手”进行的

在 `for i in range(len(nums) - 1)` 中，每一轮比较的永远是**相邻**的两个数：

- 第 1 轮：比较 `nums[0]` 和 `nums[1]`
- 第 2 轮：比较 `nums[1]` 和 `nums[2]`
- 第 3 轮：比较 `nums[2]` 和 `nums[3]`

**它不会跳格比较。** 无论你改不改 `nums[i]` 的值，下一轮 `i` 增加 1 后，比较的永远是 **此刻的** `nums[i+1]` 和 `nums[i+2]`。

------

### 2. 为什么情况 A 不用写 `nums[i] = nums[i+1]`？

**情况 A 回顾**：`[1, 4, 2, 3]` （当 `i=1` 时，`nums[i-1]=1, nums[i]=4, nums[i+1]=2`）

1. **当前轮 (`i=1`)**：发现 `4 > 2`，破坏规则。
2. **贪心决策**：因为 `nums[i+1]` (2) 比前前个元素 `nums[i-1]` (1) 大，所以我们**幻想**把 `4` 改成了 `2`。
3. **关键点**：即使你**不写** `nums[1] = 2`，在下一轮循环时：
   - `i` 变成 2。
   - 程序比较 `nums[2]` (2) 和 `nums[3]` (3)。
   - **结果**：`2 <= 3`，没有问题。
4. **结论**：因为你是把“前面”的数调小了，它**不会增加后面数的负担**。`nums[2]` 还是那个 `2`，它并没有因为 `nums[1]` 是 `4` 还是 `2` 而改变自己的大小。

------

### 3. 为什么情况 B **必须**写 `nums[i+1] = nums[i]`？

**情况 B 回顾**：`[3, 4, 2, 3]` （当 `i=1` 时，`nums[i-1]=3, nums[i]=4, nums[i+1]=2`）

1. **当前轮 (`i=1`)**：发现 `4 > 2`。
2. **贪心决策**：因为 `2` 比前面的 `3` 还要小，我们无法把 `4` 调小，只能**被迫**把 `2` 调大，改成 `4`。
3. **如果不写赋值语句会怎样？**
   - 下一轮 `i` 变成 2。
   - 程序会比较 `nums[2]` (还是原数组里的 **2**) 和 `nums[3]` (3)。
   - 程序会觉得 `2 <= 3` 是对的。
   - **结果**：程序返回 `True`。
   - **但实际上**：数组应该是 `[3, 4, 4, 3]`，最后那个 `4 > 3` 的错误被漏掉了！
4. **结论**：因为你修改的是“后面”的数（`nums[i+1]`），而这个数在下一轮循环中会作为“前数”出现。所以你**必须**真刀真枪地改掉它，否则下一轮的判断基础就错了。

------

| ---------- | -------------- | ----------------- | ------------------------------------------------------------ |
| **情况 A** | `[1, 4, 2, 5]` | 幻想把 `4` 变 `2` | 变小的是“过去”，不影响“未来”（`2` 还是小于 `5`）             |
| **情况 B** | `[3, 4, 2, 3]` | 必须把 `2` 变 `4` | 变大的是“未来”，**必须更新**，否则下一轮拿旧的 `2` 去比 `3` 就漏判了 |

### 1. 逻辑分层：谁负责“前后比较”，谁负责“前前比较”？

### 2. 为什么你觉得“没体现前后比较”？

因为当你进入 `if i == 0 or nums[i+1] >= nums[i-1]` 这个判断时，**前提条件 `nums[i] > nums[i+1]` 已经铁证如山了。**

- 你已经知道 `nums[i]` 比 `nums[i+1]` 大了（这就是前后比较的结果）。
- 现在的任务**不是**去判断它们大不大，而是**怎么改**。
- 改法取决于 `nums[i+1]` 和更早的 `nums[i-1]` 的关系。

### 3. 举例：为什么不拿 `nums[i+1]` 和 `nums[i]` 比？

假设数组是 `[1, 4, 2]`：

1. **前后比较**：发现 `4 > 2`（`nums[i] > nums[i+1]`）。**断层确认！**
2. **决策时刻**：我们要改掉 `4` 或者 `2`。
3. **如果拿 `nums[i+1]`(2) 和 `nums[i]`(4) 比**：
   - 比出来的结果肯定是 `2 < 4` 啊！这无法帮你决定是把 `4` 变小还是把 `2` 变大。
4. **拿 `nums[i+1]`(2) 和 `nums[i-1]`(1) 比**：
   - 结果是 `2 >= 1`。
   - **结论**：哪怕我把 `4` 削矮成 `2`，它依然比前面的 `1` 大。
   - **最优决策**：削矮 `4`。

### 4. 总结这个逻辑结构

- **大前提**（`nums[i] > nums[i+1]`）：确定“这里有病”。
- **小判断**（`nums[i+1] >= nums[i-1]`）：确定“怎么治病”。

**所以，并不是没体现前后比较，而是“前后比较”是进入这段决策代码的“门票”。只有门票通过了，我们才会去操心 `nums[i-1]` 的事。**

这个分层逻辑现在清晰了吗？明早练习时，你可以尝试把这几行代码读成：**“既然 `i` 比 `i+1` 大了，那么如果 `i+1` 还算争气（比 `i-1` 大），我就削弱 `i`；否则，我只能拔高 `i+1` 了。”**



Q: 如果是首对逆序，或者修改 nums[i] 就能解决问题

非常有必要。`if i == 0` 是为了处理**数组开头的特殊情况**，防止程序在尝试回头看 `nums[i-1]` 时崩溃（报错 `IndexError`），同时也涵盖了一个重要的逻辑决策。

我们可以从 **“安全性”** 和 **“逻辑正确性”** 两个角度来拆解：

### 1. 防止程序崩溃 (Index Safety)

在 Python 中，如果你访问 `nums[-1]`，它会返回数组的**最后一个元素**。

- 如果没有 `if i == 0` 的判断，当 `i = 0` 发生逆序时（比如 `[4, 2, 3]`），程序会去比较 `nums[1]` (2) 和 `nums[-1]` (3)。
- 这种“跨越时空”的比较在逻辑上是毫无意义的，甚至可能因为凑巧满足条件而误导程序。

### 2. 逻辑上的“绝对安全”

正如我们之前讨论的，**“把大的变小”永远比“把小的变大”更安全**。

当第一个元素 `nums[0]` 比第二个元素 `nums[1]` 大时：

- 我们面前没有任何障碍（没有 `nums[-1]` 需要顾虑）。
- 我们可以**无条件**地把 `nums[0]` 削减到和 `nums[1]` 一样大。
- 这样改完后，`nums[0]` 和 `nums[1]` 齐平了，且不会破坏任何“前序关系”（因为前面根本没数）。

### 3. 如果不写 `i == 0` 会发生什么？

假设数组是 `[4, 2, 1]`：

1. **i = 0**：发现 `4 > 2`。
2. **如果没有 `i == 0` 直接走 `else`**：程序去比 `nums[1]`(2) 和 `nums[-1]`(1)。
3. **结果**：`2 > 1`（不满足 `nums[i+1] >= nums[i-1]`），程序会执行 `else` 里的 `nums[i+1] = nums[i]`，即把 `2` 改成 `4`。
4. **数组变成**：`[4, 4, 1]`。
5. **下一轮**：发现 `4 > 1`，`count` 变成 2，返回 `False`。**这是对的。**

**但是！** 换个例子：`[4, 2, 3]`

1. **i = 0**：发现 `4 > 2`。
2. **如果不写 `i == 0`**：比 `nums[1]`(2) 和 `nums[-1]`(3)。
3. **结果**：`2 < 3`。程序走 `else` 把 `2` 改成 `4`。
4. **数组变成**：`[4, 4, 3]`。
5. **下一轮**：发现 `4 > 3`，返回 `False`。
6. **错误！** 这个数组明明改一次就能变好（改成 `[2, 2, 3]`），结果你因为没写 `i == 0` 导致决策错误，判定成了 `False`。

### 💡 结论

`i == 0` 是一个**“绿色通道”**：

- 它告诉程序：“既然这是第一块砖，后面那块砖即使再矮，我也能通过削矮第一块砖来对齐，而不需要担心踢到前面的砖。”

**所以，这行代码是逻辑严密性的保障。**



Q: nums[-1]是最后一个元素吗

是的，在 Python 中，**`nums[-1]` 确实代表数组（列表）的最后一个元素**。

这是 Python 的 **负索引（Negative Indexing）** 特性。你可以这样记忆：

- `0` 是第一个元素。
- `-1` 是**末尾算起的第一个**元素。
- `-2` 是末尾算起的第二个元素，以此类推。

### 为什么在 [Problem 4: Non-decreasing Array](https://courses.codepath.org/courses/tip103/unit/1#session_one) 中这很重要？

正因为 Python 支持负索引，如果你在代码中写了 `nums[i-1]` 而没有用 `if i == 0` 来保护，程序**并不会报错**，但会产生逻辑错误：

1. **静默错误**：当 `i = 0` 时，`nums[i-1]` 实际上变成了 `nums[-1]`（最后一个数）。
2. **错误比对**：你会莫名其妙地拿数组开头的数去和数组末尾的数做比较。
3. **判定失效**：这会导致你的“贪心决策”基于错误的信息，从而给出错误的 `True` 或 `False`。

------

Python

```
nums = [10, 20, 30, 40]

print(nums[-1])  # 输出 40 (最后一个)
print(nums[-2])  # 输出 30 (倒数第二个)

# 如果你在循环里没注意：
i = 0
```

### Problem 5: Missing Clues (缺失的线索)

**题意**：给定一个范围 `[lower, upper]` 和一个已经存在的线索列表 `clues`，找出这个范围内所有缺失的数字，并以“区间”的形式返回。

- **Understand**：
  - 输入：`clues = [0, 1, 3, 50, 75]`, `lower = 0`, `upper = 99`
  - 输出：`[[2, 2], [4, 49], [51, 74], [76, 99]]`
  - 注意：如果缺失的是单个数字，区间就是 `[x, x]`。
- **Plan**：
  1. 为了方便处理，先将 `clues` 排序。
  2. 设定一个指针 `next_expected = lower`，代表我们目前期待看到的最小数字。
  3. 遍历 `clues` 中的每个数字 `c`：
     - 如果 `c > next_expected`，说明 `next_expected` 到 `c - 1` 这一段都丢了。记录区间 `[next_expected, c - 1]`。
     - 更新 `next_expected = c + 1`。
  4. **收尾工作**：循环结束后，如果 `next_expected <= upper`，说明最后还有一段缺失，记录 `[next_expected, upper]`。
- **Implementation**：

Python

```python
def find_missing_clues(clues, lower, upper):
    clues.sort()
    missing_ranges = []
    next_expected = lower
    
    for c in clues:
        if c > next_expected:
            missing_ranges.append([next_expected, c - 1])
        next_expected = c + 1
    
    if next_expected <= upper:
        missing_ranges.append([next_expected, upper])
        
    return missing_ranges
```

### Problem 6: Vegetable Harvest (蔬菜收割)

**题意**：给定一个 2D 矩阵（菜地），如果格子里是 `'c'`（代表胡萝卜），就计数。统计总共有多少个胡萝卜。

  - 这是一个最基础的**二维数组遍历**。

  1. 初始化计数器 `count = 0`。

Python

```python
def harvest(vegetable_patch):
    count = 0
    for row in vegetable_patch:
        for item in row:
            if item == 'c':
                count += 1
    return count
```

### Problem 7: Eeyore's House (屹耳的房子)

**题意**：有两个木棍堆 `pile1` 和 `pile2`。给定系数 `k`。如果 `pile1[i]` 能被 `pile2[j] * k` 整除，这就叫一个“好对子”。求总共有多少对。

- **Understand**：
  - “能被整除”意味着余数为 0，Python 语法是 `a % b == 0`。
- **Plan**：
  1. 初始化 `count = 0`。
  2. 双重循环：
     - 外层取出 `pile1` 的每个木棍 `s1`。
     - 内层取出 `pile2` 的每个木棍 `s2`。
  3. 计算分母 `denominator = s2 * k`。
  4. 判断 `if s1 % denominator == 0`。
- **Implementation**：

Python

```
def good_pairs(pile1, pile2, k):
    count = 0
    for s1 in pile1:
        for s2 in pile2:
            if s1 % (s2 * k) == 0:
                count += 1
    return count
```

这道题目 [Island Perimeter (岛屿周长)](https://leetcode.com/problems/island-perimeter/) 是 LeetCode 上一道非常经典的 **Easy** 难度题目，属于 **Arrays (数组)** 和 **Matrix (矩阵)** 类型。

它的核心逻辑其实非常直观，不需要用到复杂的算法（如 DFS/BFS），仅通过基础的**迭代（Iterative）**就可以高效解决。

------

### 1. 题目大意

给定一个 $n \times m$ 的二维网格 `grid`，其中 `1` 代表陆地，`0` 代表水域。

- 网格中恰好只有**一个**岛屿。
- 岛屿内部没有“湖泊”（即岛屿内没有被陆地完全包围的水域）。
- 每个格子是边长为 1 的正方形。
- 你需要计算这个岛屿的**总周长**。

------

### 2. 核心解题思路（两种方法）

#### 方法 A：减法逻辑（最推荐）

这是最简单且最容易理解的方法：

1. **假设每块陆地都是独立的**：如果一块陆地周围全是水，它的周长贡献是 **4**。
2. **处理相邻关系**：如果两块陆地相邻，它们接触的那条边就**消失**了，不再是周长。
   - 关键点：每有一对相邻的陆地，总周长就要 **减去 2**（因为这条重合边本属于两块陆地）。

**算法步骤：**

- 遍历整个矩阵。
- 如果当前格子是陆地 (`grid[i][j] == 1`)：
  - 周长基础值 `ans += 4`。
  - 检查它的**上方**是否也是陆地。如果是，`ans -= 2`。
  - 检查它的**左边**是否也是陆地。如果是，`ans -= 2`。
- **为什么只检查左和上？** 因为我们是从左往右、从上往下遍历的，这样可以确保每对相邻关系只被计算（和扣除）一次。

#### 方法 B：直接计数逻辑

1. 遍历每一块陆地。
2. 观察它的上下左右四个方向。
3. 如果某个方向是**水域**或者**超出了边界**，周长就加 **1**。

------

### 3. 代码实现 (Python)

Python

```
class Solution:
    def islandPerimeter(self, grid: List[List[int]]) -> int:
        rows = len(grid)
        cols = len(grid[0])
        perimeter = 0
        
        for r in range(rows):
            for c in range(cols):
                if grid[r][c] == 1:
                    # 基础周长
                    perimeter += 4
                    
                    # 检查左边是否有邻居
                    if c > 0 and grid[r][c - 1] == 1:
                        perimeter -= 2
                        
                    # 检查上边是否有邻居
                    if r > 0 and grid[r - 1][c] == 1:
                        perimeter -= 2
                        
        return perimeter
```

- **空间复杂度**：$O(1)$。我们只需要一个变量来存储结果。

不是的，`if grid[r][c] == 1` 这一行代码本身**并不是**指周长，它是一个**判断条件**，意思是：

**“如果当前这个格子是陆地”**。

在 [Island Perimeter (LeetCode 463)](https://leetcode.com/problems/island-perimeter/) 这道题里，我们可以这样拆解这部分逻辑：

### 1. 它是如何计算周长的？

当我们确定一个格子是陆地（即满足 `grid[r][c] == 1`）时，我们才会开始计算它的周长贡献：

- **初始贡献**：我们先假设这块陆地是孤立的，所以给 `perimeter` 直接加 **4**（因为它有上下左右四条边）。
- **后续修正**：接着我们会检查它的邻居。如果发现它的左边或上边也是陆地，就说明这两块陆地“贴在了一起”。一旦贴在一起，原本属于这两块陆地的**两条边就重合消失了**，所以我们要从总数里**减去 2**。

### 2. 代码逻辑翻译

把代码翻译成大白话就是：

Python

```
if grid[r][c] == 1:            # 如果我踩在了一块陆地上
    perimeter += 4             # 先给它算上 4 条边
    
    if c > 0 and grid[r][c-1] == 1: # 如果我左边也是陆地
        perimeter -= 2         # 我们俩共用一条边，所以总周长要减去 2
        
    if r > 0 and grid[r-1][c] == 1: # 如果我上边也是陆地
        perimeter -= 2         # 我们俩也共用一条边，总周长再减去 2
```

- **一个积木**：周长是 4。
- **两个积木分开放**：总周长是 4 + 4 = 8。
- **两个积木并排贴在一起**：它们中间接触的那两条边“藏起来”了。现在它们的总周长变成了 8 - 2 = **6**。

让我们换个视角看这段代码的操作流程：

### 1. 默认“孤岛”假设

代码中 `perimeter += 4` 的意思是：**“管它周围有没有水，只要这一格是陆地，我先预支给它 4 条边。”**

这时候，这块陆地的状态在程序心里是这样的：

### 2. 只有“邻居”能收回这 4 条边

为什么代码里没有 `if grid[r+1][c] == 0`（判断周围是水）？因为：

- **如果周围真的是水**：那么 `grid[r][c-1]` 或 `grid[r-1][c]` 的判断条件就不会成立。
- **结果**：没有任何 `if` 被触发，没有任何 `-= 2` 被执行。
- **最终贡献**：这块陆地最后留在总数里的贡献就是初始的 **4**。

**所以，“周围全是水”这个信息，是通过“不满足扣分条件”来隐式体现的。**

------

### 3. 为什么只扣左边和上边？

你可能会问：“那右边和下边有邻居怎么办？不用扣吗？”

这是为了**防止重复计算**。

- 当你处理 `grid[r][c]` 时，如果它的**右边**有陆地，你现在不扣。
- **但是**，当循环走到右边那一格时，对于那一格来说，现在的 `grid[r][c]` 就是它的**左边**邻居。
- **那时候**，程序会触发 `if c > 0 and grid[r][c-1] == 1`，在那一刻把这两块陆地重合的 **2 条边**扣掉。

------

### 4. 逻辑对比：正面判断 vs 倒扣逻辑

| **逻辑类型**        | **思路**             | **代码体现**                                                 |
| ------------------- | -------------------- | ------------------------------------------------------------ |
| **正面判断**        | 只有旁边是水才加 1   | `if r == 0 or grid[r-1][c] == 0: res += 1` (需判断 4 个方向) |
| **倒扣逻辑 (本题)** | 先给 4，有邻居就扣 2 | `res += 4; if neighbor: res -= 2` (只需判断 2 个方向)        |

**倒扣逻辑的好处**：代码更简洁，且不容易在处理矩阵边界（比如第一行、第一列）时写出复杂的 `if-else`。

这道题 [1672. Richest Customer Wealth](https://leetcode.com/problems/richest-customer-wealth/) 是 LeetCode 上非常经典的一道 **Easy** 题目，也正是 [Problem Set Version 1](https://courses.codepath.org/courses/tip103/unit/1#session_one) 中 **Problem 5: Heist (Version 2)** 的原型。

如果你已经理解了前面的蔬菜收割（Problem 6），这道题其实就是它的“升级版”。

------

### 1. 题目大意

给你一个 $m \times n$ 的整数网格 `accounts`，其中：

- `accounts[i]` 代表第 `i` 位客户。
- `accounts[i][j]` 代表第 `i` 位客户在第 `j` 家银行存的钱。

**任务**：算出每位客户在所有银行的存款总和（即总资产），并返回**最富有**的那位客户拥有的资产数额。

------

### 2. 解题思路

这道题的核心是 **2D Array Traversal（二维数组遍历）**，但比周长那题简单，因为它不需要考虑邻居关系，只需要累加。

1. **外层循环**：遍历每一位客户（即每一行）。
2. **内层操作**：
   - 计算该客户当前所有银行存款的总和 `sum(customer_accounts)`。
3. **全局更新**：
   - 维护一个变量 `max_wealth`，如果当前客户的资产比 `max_wealth` 大，就更新它。

------

### 3. 代码实现

#### 方法一：最直观的嵌套循环

这是最能体现底层逻辑的写法，适合初学者掌握 $O(m \times n)$ 的遍历。

Python

```
def maximumWealth(accounts):
    max_wealth = 0
    # 遍历每一位客户
    for customer in accounts:
        current_customer_wealth = 0
        # 累加该客户在所有银行的钱
        for money in customer:
            current_customer_wealth += money
        
        # 更新最大值
        if current_customer_wealth > max_wealth:
            max_wealth = current_customer_wealth
            
    return max_wealth
```

#### 方法二：利用 Python 内置函数（推荐）

在面试中，如果你展示出这种简洁的写法，会显得你对语言非常熟练。

Python

```
def maximumWealth(accounts):
    # 用 max() 找出一组总和里的最大值
    return max(sum(customer) for customer in accounts)
```

### ❌ 错误检查

1. **`for j in (len(accounts[i]))` 语法错误**：
   - `len()` 返回的是一个整数（比如 3），你不能直接对一个整数进行 `in` 循环。
   - **修正**：必须加上 `range()`，写成 `for j in range(len(accounts[i]))`。
2. **`array_sum` 累加逻辑错误**：
   - 在你的代码中，`array_sum = 0` 是在循环最外面定义的。这意味着当你计算完第一个人的钱，开始算第二个人的钱时，`array_sum` 会在第一个人的基础上继续累加，而不是重新清零。
   - **结果**：最后返回的其实是全班人的钱总和，而不是那个最富有的人。
   - **修正**：每进入一个新客户（外层循环）时，都要把 `array_sum` 重置为 `0`。
3. **变量初始化**：
   - 虽然在这道题（存款都是正数）里没问题，但习惯上，如果 `accounts` 可能为空，`max_sum` 设为 `0` 是安全的。



# 题3162: find the number of good pairs

```python
# @lc code=start
# 题意：有两个木棍堆 pile1 和 pile2。给定系数 k。如果 pile1[i] 能被 pile2[j] * k 整除，这就叫一个“好对子”。求总共有多少对。
class Solution:
    def numberOfPairs(self, nums1: List[int], nums2: List[int], k: int) -> int:
        count = 0
        for s1 in nums1:
            for s2 in nums2:
                if s1 % (s2 * k) == 0: # “能被整除”意味着余数为 0，Python 语法是 a % b == 0
                    count += 1
        return count
```
